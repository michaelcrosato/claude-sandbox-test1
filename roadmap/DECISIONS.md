# Decisions Log (append-only, ADR-lite)

> One entry per autonomous judgment call: context → decision → reversible? → where it lives.

- QA surface: charter names Docker/Fly.io (single-container, no Redis) → adopted Docker/Fly.io as the deployment and QA surface; persistent staging from `develop` on Fly.io; preview per PR (department)
- Database: charter specifies SQLite via node:sqlite (built-in, no external service) as the default, with optional PostgreSQL backend for scale-out → adopted SQLite as the primary database; no external DB CLI needed (department)
- E2E approach: charter specifies vitest + compiled-dist smoke tests covering ~2060 tests across all routes and wire contracts → adopted vitest compiled-dist smokes as the E2E verification method (department)
- GitHub repo: michaelcrosato/claude-sandbox-test1 (as specified by department runbook) (department)
- Package manager / toolchain: npm, node ≥22.13, tsc, biome, vitest — all confirmed by existing package.json and charter docs (department)
- F-0001 verification lane: local PowerShell can run Node 24 and product checks, but Git Bash lacks a native `node` binary and the Windows `node.exe` path/env bridge breaks existing hook fixtures → use Ubuntu CI as the authoritative full `verify.sh` evidence for this branch; reversible by installing native Node in the Bash environment.
- F-0007 endpoint signing secrets: delivery requires recovering endpoint secrets, while F-0004 had only a digest → store new endpoint secrets with local AES-GCM protection and key metadata in SQLite; keep digest-only legacy rows fail-closed; reversible by adding external KMS/configured key support later.
- F-0007 outbound delivery safety: receiver redirects and response bodies are not needed for webhook semantics → disable automatic fetch redirects and cancel response bodies after headers/status; reversible if a future feature adds explicit redirect allowlists or response-body auditing.
- F-0009 attempt pagination: message attempt logs can grow without a natural offset cap → use bounded keyset pagination by `attempted_at DESC, id DESC`; reversible if the public API later standardizes a different cursor envelope.
- F-0010 admin route auth ordering: enabled admin routes should not reveal method support to invalid tokens → authenticate once before method dispatch, while still returning `404` before auth when admin is disabled; reversible only with a documented public route-discovery policy.
- F-0011 quota and billing period semantics: producer retries should not be punished for a message that was already accepted, and month boundaries need one deterministic definition → enforce quota only for new accepted messages after the idempotent replay check, and key usage by UTC `YYYY-MM`; reversible with a documented billing-period migration.
- F-0012 batch contract: README implied array input and `batch.results` but not extra metadata → use a top-level JSON array and `{ results: [{ ok, message, fanout } | { ok, error }] }` without per-item indexes or summaries; reversible by additive fields in a future API contract feature.
- F-0013 API contract source of truth: planned README routes and runtime routes were mixed in one table → generate OpenAPI from a code-owned implemented-route list plus a closed runtime error-code enum, and mark README rows as `implemented` or `planned`; reversible by moving the same source of truth to generated docs later.
- F-0014 client honesty: SDK acceptance included message status and retry while those routes were still planned → implement `GET /v1/messages/:id` and `POST /v1/messages/:id/retry` first, then expose SDK/CLI methods only for live OpenAPI routes; reversible by expanding the same route map as future client surfaces land.
- F-0015 metrics shape: Prometheus metrics must be useful without exposing tenant-specific data → compute aggregate metrics from durable SQLite tables at scrape time and limit labels to `outcome`, `status`, normalized `reason`, and `version`; reversible by adding append-only metric counters if strict monotonicity across tenant deletion is required.
- F-0016 container defaults: Docker support should not expose unauthenticated operator surfaces or local SQLite state by accident → bind Compose ports to `127.0.0.1`, require admin token interpolation, and exclude env/key/database files from build context; reversible with a documented production proxy/firewall feature.
- F-0017 diagnostic sends: endpoint test-send should exercise real receiver behavior without becoming message intake → send only to the stored endpoint URL, sign the same webhook body format, report synchronous result fields, and avoid durable message/delivery/usage mutation; reversible if a future audit feature adds explicit test-send history.
- F-0018 tenant history: the tenant dashboard needs real debugging history, not browser-local recent lookups → implement `GET /v1/messages` with tenant-scoped newest-first keyset pagination and keep richer filters planned; reversible by adding filter parameters without changing the base response envelope.
- F-0019 secret rotation semantics: receiver cutover needs zero downtime without exposing old secrets indefinitely → rotate endpoint secrets in-place, keep only the previous encrypted secret for a bounded overlap window, sign delivery and test-send webhooks with both secrets during overlap, and fail closed on corrupt active previous-secret metadata; reversible by adding a multi-version secret history table later.
- F-0020 endpoint observability scope: receiver debugging should not require exposing payloads or endpoint configuration → return endpoint-scoped delivery metadata and aggregate stats only, with bounded pagination/windows and no payload, URL, header, API-key, signing-secret, protected-secret, or response-body fields; reversible by adding separate privileged detail routes later.
- F-0021 app-wide delivery search scope: operators need delivery search without message ids, but broad logs can leak sensitive context → return only tenant-scoped delivery metadata, use bounded filters/pagination, match failure reason through latest delivery error or attempt records, and omit payloads, endpoint URLs/headers, API keys, signing secrets, protected secret fields, request bodies, and response bodies; reversible by adding separate privileged detail views later.
- F-0022 Python SDK redirect policy: API clients should not trade convenience for token safety → reject HTTP redirects instead of following them with bearer credentials, and surface the redirect as a normal `PosthornApiError`; reversible only with an explicit same-origin redirect policy and regression tests.
- F-0023 SDK query construction: typed helper inputs reduce mistakes, but JavaScript callers can still pass extra fields → serialize only per-helper allowlisted query keys so secret-like accidental properties never enter URLs; reversible only if a future query builder has equivalent allowlist behavior.
- F-0024 admin SDK boundary: operator credentials and tenant credentials have different blast radii → expose a separate `PosthornAdminClient` that sends the admin token only as a bearer header, while API key secrets remain one-time create responses and list helpers return metadata only; reversible only with equivalent credential separation and secret redaction tests.
- F-0025 auto-disable guard: endpoint disabling should require sustained failure evidence, not a single slow/current dead-letter → count only failed/dead-letter attempts older than the current attempt and require no success after the cutoff; reversible only with an equivalent single-failure DoS regression.
- F-0026 message filter timestamps: history windows need deterministic SQLite text comparisons → accept only calendar-valid date-times with explicit `Z` or numeric offset, normalize to UTC ISO strings, and reject local-time or rollover dates; reversible only with equivalent deterministic validation.
- F-0027 app system secret rotation: app-level signing secrets are operational credentials and should behave like endpoint signing secrets without leaking through admin reads → store current and previous protected app system secrets on `apps`, return each raw `whsec_` only once from the rotate command, and perform rotation in one immediate SQLite transaction; reversible only with equivalent protected storage, one-time output, and atomicity regression coverage.
- F-0028 CLI error redaction: command-line mistakes can include pasted credentials → admin and tenant CLI usage errors must not reflect unknown raw argv values, and option-looking admin key names are rejected before API calls; reversible only with equivalent token/one-time-secret stderr regression coverage.
- F-0029 monitoring artifact boundaries: dashboards and alerts must not invent product telemetry or expose tenant dimensions → publish starter artifacts that reference only implemented aggregate metrics and Prometheus `up`, with tests rejecting unknown `posthorn_*` metric names; reversible by adding new metric names and artifact tests together.
- F-0030 endpoint throttling state: delivery claim throttles must not scan unbounded attempt history while holding SQLite's write lock → store per-endpoint internal claim-window state (`rate_limit_window_started_at`, `rate_limit_window_count`) and update it atomically with leases; reversible only with an equivalent bounded/indexed throttle algorithm and starvation regressions.
