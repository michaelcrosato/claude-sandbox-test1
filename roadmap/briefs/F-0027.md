# Builder Brief — F-0027: Admin app system secret rotation

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped because this is the last README-planned API route: `POST /v1/admin/apps/:id/rotate-system-secret`.

## Feature

Implement app-level system signing secret rotation for operators. The route should mint a new Standard Webhooks `whsec_` secret, store it protected at rest, keep the previous protected secret for a bounded overlap window, and return the new raw secret exactly once.

This feature creates the protected storage and control-plane surface only. It does not add any new app-level outbound webhook delivery behavior.

## Spec

Add nullable columns to `apps` for protected app system signing secrets:

- `system_signing_secret_ciphertext`
- `system_signing_secret_key_version`
- `system_signing_secret_nonce`
- `previous_system_signing_secret_ciphertext`
- `previous_system_signing_secret_key_version`
- `previous_system_signing_secret_nonce`
- `previous_system_signing_secret_expires_at`

Update `initializeSchema()` with a migration for these columns. Existing apps have null current-secret columns and must still be rotatable.

Add admin helper:

```ts
rotateAdminAppSystemSecret(storage, appId, input?, now?)
```

Input:

- Optional JSON object.
- `overlapSeconds?: number`
- Use the same bounds as endpoint rotation: integer 60 through 2,592,000 seconds (30 days).
- Default overlap should be 86,400 seconds (24 hours).

Behavior:

- Missing app returns `null`.
- Generate a new `whsec_` secret with existing Standard Webhooks utilities.
- Store the new secret with existing local AES-GCM secret protection.
- If the app had a current protected secret, copy it into previous-system-secret columns and set `previousSystemSecretExpiresAt = now + overlapSeconds`.
- If the app had no current protected secret, leave previous-system-secret columns null.
- Return `{ app, secret, previousSecretExpiresAt }`, where `previousSecretExpiresAt` is `null` for legacy/no-current-secret apps.
- Do not expose raw secret, ciphertext, nonce, key version, or previous-secret metadata in `AdminAppRecord`, app read/list/update responses, dashboards, metrics, or errors.

Add HTTP route:

- `POST /v1/admin/apps/:id/rotate-system-secret`
- Admin routes remain disabled with 404 when `POSTHORN_ADMIN_TOKEN` is unset.
- Admin auth must happen before method dispatch, matching existing admin behavior.
- `GET`/other methods on this path with valid admin auth return 405.
- Missing app returns 404.

Add TypeScript admin client helper:

- `rotateAppSystemSecret(appId, input?: { overlapSeconds?: number })`
- Add route mapping coverage in `POSTHORN_ADMIN_CLIENT_ROUTES`.

Update OpenAPI and README:

- Mark the route implemented.
- Add response schema fields for `app`, `secret`, and nullable `previousSecretExpiresAt`.
- Show a short admin SDK example, but make clear the raw secret is shown once.

## Acceptance criteria (verbatim from roadmap/features.json)

1. POST /v1/admin/apps/:id/rotate-system-secret is implemented, disabled when POSTHORN_ADMIN_TOKEN is unset, admin-authenticated before method dispatch, and returns 404 for missing apps.
2. The route creates a new whsec_ system signing secret, stores it protected at rest, preserves the previous protected secret only for a configurable overlap window, and returns the new secret exactly once without exposing it in app read/list/update responses.
3. Existing apps with no stored system secret can rotate successfully by creating the first protected system secret without previous-secret overlap metadata.
4. OpenAPI, TypeScript admin client route mappings/helpers, README API/admin SDK docs, and route coverage reflect the implemented admin route.
5. Tests cover success, missing app, disabled admin routes, admin auth/method behavior, protected storage/no secret leakage, and legacy app rotation.
