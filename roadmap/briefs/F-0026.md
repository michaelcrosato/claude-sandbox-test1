# Builder Brief — F-0026: Message history filters

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped because this is a direct README contract gap: `GET /v1/messages` is implemented, but README still marks `eventType`, `channel`, and created-at filters as planned. This feature implements the useful current-schema filters only: `eventType`, `after`, and `before`.

## Feature

Let tenants filter message history server-side by event type and created-at windows. This keeps history browsing and support workflows bounded without forcing clients to fetch every message and filter locally.

## Spec

Update `GET /v1/messages` so it accepts these query parameters:

- `limit`
- `cursor`
- `eventType`
- `after`
- `before`

Filter semantics:

- `eventType`: exact match on `messages.event_type`; validate with the existing event type identifier rules.
- `after`: include messages with `created_at >= after`.
- `before`: include messages with `created_at < before`.
- `after` and `before` must be valid date-time strings. Normalize accepted date strings to `new Date(value).toISOString()` before comparing so the SQLite text comparison stays deterministic.
- If both `after` and `before` are supplied, reject `after >= before` with `invalid_request`.
- Filters combine with keyset pagination and preserve newest-first ordering by `created_at DESC, id DESC`.
- The cursor remains the existing `{ createdAt, id }` cursor. Do not encode filters into the cursor; clients must pass the same filters with subsequent pages.

Update the TypeScript client:

- `ListMessagesInput` should include typed `eventType?: string`, `after?: string`, and `before?: string`.
- `PosthornClient.listMessages()` must serialize only `eventType`, `after`, `before`, `limit`, and `cursor`. Keep the query allowlist protection; do not allow arbitrary extra keys.

Update OpenAPI and README:

- Add OpenAPI parameter docs for `GET /v1/messages` covering the five supported query parameters.
- Update README API table and TypeScript SDK sample so message filters are documented as implemented.
- Do not implement or document a `channel` filter; there is no message channel column.

## Acceptance criteria (verbatim from roadmap/features.json)

1. GET /v1/messages supports eventType, after, and before query parameters in addition to limit and cursor, with eventType using the existing event type identifier validation and after/before requiring valid date-time strings.
2. Message history filters are tenant-scoped, combine with keyset pagination, preserve newest-first ordering, and do not expose idempotency keys, payload hashes, API keys, endpoint secrets, or protected secret metadata.
3. The TypeScript PosthornClient listMessages helper exposes typed eventType, after, and before inputs and serializes only the allowlisted message-history query keys.
4. OpenAPI parameters and README route/SDK documentation reflect the implemented message filter surface and no longer describe those filters as planned.
5. Tests cover eventType filtering, after/before created-at windows, filter pagination, tenant isolation, validation failures, and SDK query allowlisting.
