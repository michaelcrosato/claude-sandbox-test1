# Builder Brief - F-0006: Core message intake and fanout queue

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped; F-0004 already established the auth, gateway, and endpoint-store patterns this feature must extend.

## Feature
Implement `POST /v1/messages` from `README.md#api-routes`. Authenticated producers can submit an event, Posthorn persists the message, and the store creates pending delivery work for every enabled tenant endpoint whose event filter matches.

## Spec
Current state:

- `src/auth.ts` authenticates tenant `phk_` API keys and returns `appId`.
- `src/endpoints.ts` persists tenant endpoints with `event_types_json` filters and `enabled`.
- `src/gateway.ts` already has JSON body reading, request size caps, error envelopes, and `/v1/endpoints` routing.
- `src/storage.ts` already creates `messages` and `deliveries` tables.

Implement message intake:

- Add `src/messages.ts` with no runtime dependencies.
- Export `acceptMessage(storage, appId, input, now?)`.
  - Validate `input` is a JSON object.
  - Require `eventType` as a non-empty full-stop-delimited identifier made from letters, digits, and underscores.
  - Require `payload` to be present, JSON-serializable, and not `null`.
  - Generate `msg_` message IDs and `del_` delivery IDs.
  - Persist one row in `messages` with `app_id`, `event_type`, canonical `payload_json`, and `created_at`.
  - Fan out to enabled endpoints for the same tenant:
    - `event_types_json` null means match all event types.
    - otherwise match when the submitted `eventType` is included in the endpoint filter array.
    - disabled endpoints do not get delivery rows.
  - Persist one `deliveries` row per match with `status = 'pending'`, `attempt_count = 0`, `created_at`, and `updated_at`.
  - Return `{ message, fanout }`.
- Export `getMessage(storage, appId, messageId)` and `listDeliveriesForMessage(storage, appId, messageId)` for tests and later routes.
- Message response shape:
  - `message`: `id`, `eventType`, `payload`, `createdAt`.
  - `fanout`: `matched`, `deliveryIds`, `endpointIds`.
- Add `MessageValidationError` with stable `code = 'invalid_request'`.
- Wire `POST /v1/messages` into `src/gateway.ts`.
  - Authenticate with the same bearer API-key helper as endpoint routes.
  - Return `202 { "message": ..., "fanout": ... }` on success.
  - Missing/invalid auth returns `401` with `unauthorized`.
  - Malformed JSON returns `400` with `invalid_json`.
  - Validation failures return `400` with `invalid_request`.
  - Unsupported methods on `/v1/messages` return `405`.
- Re-export public message types/helpers from `src/index.ts`.
- Update README implementation-status wording to mention message intake and fanout queue.

Testing:

- Add HTTP integration tests using an in-memory gateway and synthetic tenants/API keys.
- Seed tenants and endpoint rows through existing helpers/SQL.
- Cover missing auth and revoked/invalid auth.
- Cover validation failures: missing eventType, invalid eventType, missing payload, null payload, malformed JSON.
- Cover accepted message persistence and readback through `getMessage`.
- Cover no-match fanout: accepted message, `matched = 0`, no delivery rows.
- Cover one-match fanout: exactly one pending delivery for a matching enabled endpoint.
- Cover multi-endpoint fanout: multiple matching enabled endpoints get pending delivery rows.
- Cover disabled endpoints and other-tenant endpoints do not receive delivery rows.

## Acceptance criteria (verbatim from roadmap/features.json)
1. POST /v1/messages validates eventType and payload, authenticates a tenant API key, and returns 202 for accepted messages.
2. Accepted messages are persisted with created-at metadata and can be read back through the store.
3. Fanout creates one pending delivery task for each enabled endpoint whose filter matches the event type.
4. Tests cover authentication, validation errors, no-match fanout, one-match fanout, and multi-endpoint fanout.
