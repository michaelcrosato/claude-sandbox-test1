# Builder Brief — F-0024: TypeScript admin client

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped because this is a direct README contract gap: `README.md#admin--control-plane-sdk` documents `PosthornAdminClient`, but no such export exists.

## Feature

Implement the README-documented TypeScript admin/control-plane client. Operators should be able to provision tenants, manage quotas, mint/list/revoke tenant API keys, and read billing usage through the package without hand-writing admin HTTP requests. This must use the existing admin HTTP routes and the same error semantics as the tenant SDK.

## Spec

Update `src/client.ts` and exports as needed.

Add `PosthornAdminClient` with constructor:

```ts
new PosthornAdminClient({ baseUrl, adminToken, fetch? })
```

It should normalize `baseUrl`, require a non-empty `adminToken`, send `Authorization: Bearer <adminToken>`, and reuse `PosthornApiError` behavior for non-2xx responses.

Required admin helpers:
- `createApp(input: { name: string; monthlyMessageQuota?: number | null })`
- `listApps()`
- `getApp(appId: string)`
- `updateApp(appId: string, input: { name?: string; monthlyMessageQuota?: number | null })`
- `deleteApp(appId: string)`
- `getAppUsage(appId: string)`
- `createApiKey(appId: string, input?: { name?: string | null })`
- `listApiKeys(appId: string)`
- `revokeApiKey(keyId: string)`

Return the gateway's existing JSON shapes. `deleteApp` and `revokeApiKey` return `Promise<void>` on `204`.

Add `POSTHORN_ADMIN_CLIENT_ROUTES` route mappings and coverage in `tests/client.test.ts` against `IMPLEMENTED_ROUTES`. Do not weaken `POSTHORN_CLIENT_ROUTES` coverage.

Add admin client tests in `tests/client.test.ts` using an in-process gateway with `adminToken` set:
- create/list/get/update/delete app
- create/list/revoke API key
- read app usage after seeding or sending a message through the minted tenant key
- one stable admin error path, for example invalid admin token returns `PosthornApiError` with `401` and `unauthorized`

Update `README.md` Admin / Control-Plane SDK section so it is truthful for the implemented methods only. Do not mention planned system secret rotation.

## Acceptance criteria (verbatim from roadmap/features.json)

1. The package exports PosthornAdminClient with typed helpers for app create/list/read/update/delete, app usage, API key create/list, and API key revoke.
2. Admin client helpers authenticate with POSTHORN_ADMIN_TOKEN bearer credentials and reuse PosthornApiError behavior for non-2xx responses.
3. Admin client route mappings are covered against implemented OpenAPI admin routes without weakening existing tenant SDK route coverage.
4. Client tests exercise representative admin helper success paths and at least one stable admin error path against the in-process gateway with admin routes enabled.
5. README Admin / Control-Plane SDK documentation reflects the implemented helper surface and avoids claiming unimplemented admin routes are available.
