# Builder Brief — F-0022: Python SDK and webhook verification helper

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped because the next feature came from direct repo inspection plus live competitor research: the README already promises `clients/python`, but that package does not exist; Svix currently treats Python as a first-class official SDK surface.

## Feature

Ship the README-promised Python SDK from `README.md#python-sdk`. Python producers should be able to call Posthorn without hand-writing HTTP requests, and Python receivers should be able to verify Standard Webhooks signatures with the same contract as the TypeScript helper. Keep the package dependency-free and test it from the existing Node/Vitest harness so the repository still has one primary gate.

## Spec

Create a Python 3.9+ package under `clients/python`.

Required files:
- `clients/python/README.md`
- `clients/python/posthorn/__init__.py`
- `clients/python/posthorn/client.py`
- `clients/python/posthorn/webhooks.py`
- `clients/python/pyproject.toml`

The public Python API must export:
- `PosthornClient`
- `PosthornApiError`
- `PosthornError`
- `WebhookVerificationError`
- `verify_webhook`

`PosthornClient(base_url: str, api_key: str, timeout: float = 10.0)` must use only Python standard library modules. Normalize trailing slashes from `base_url`. Every request must send `Authorization: Bearer <api_key>`, `Accept: application/json`, and JSON content type for JSON bodies. JSON response parsing should return Python dictionaries/lists matching the gateway response. Empty `204` responses should return `None`.

Client error behavior:
- Raise `PosthornApiError(status, code, message, body)` for non-2xx gateway responses.
- `code` and `message` must come from `{ "error": { "code": "...", "message": "..." } }` when present.
- If the body is not that envelope, use `code="http_error"` and a concise fallback message.
- Network/serialization failures can raise `PosthornError`.

Client methods:
- `create_endpoint(url, event_types=None, headers=None) -> dict` maps to `POST /v1/endpoints`.
- `list_endpoints() -> list` maps to `GET /v1/endpoints`.
- `send_message(event_type, payload, idempotency_key=None) -> dict` maps to `POST /v1/messages`.
- `send_message_batch(items) -> dict` maps to `POST /v1/messages/batch`; pass item dictionaries through.
- `get_message(message_id) -> dict` maps to `GET /v1/messages/:id`.
- `retry_message(message_id) -> dict` maps to `POST /v1/messages/:id/retry`.
- `list_message_attempts(message_id, limit=None, cursor=None) -> dict` maps to `GET /v1/messages/:id/attempts`.
- `get_usage() -> dict` maps to `GET /v1/usage`.
- `list_endpoint_deliveries(endpoint_id, limit=None, cursor=None) -> dict` maps to `GET /v1/endpoints/:id/deliveries`.
- `get_endpoint_stats(endpoint_id, days=None) -> dict` maps to `GET /v1/endpoints/:id/stats`.
- `list_deliveries(status=None, endpoint_id=None, event_type=None, failure_reason=None, limit=None, cursor=None) -> dict` maps to `GET /v1/deliveries`.

Use `urllib.parse.urlencode` for query strings and omit parameters whose value is `None`. Do not print secrets or log request bodies.

`verify_webhook(secret, headers, body, tolerance_seconds=300, now=None)` must:
- Accept `headers` as a normal mapping with case-insensitive lookup for `webhook-id`, `webhook-timestamp`, and `webhook-signature`.
- Accept `body` as `bytes` or `str`; sign bytes exactly, encoding strings as UTF-8.
- Accept `whsec_` secrets encoded with URL-safe base64 without padding, matching the TypeScript implementation.
- Compute HMAC-SHA256 over `"{id}.{timestamp}.{raw_body}"`.
- Accept Standard Webhooks multi-signature headers such as `v1,<base64> v1,<base64>` and succeed when any `v1` signature matches.
- Use constant-time comparison.
- Reject missing headers, malformed timestamps, stale/future timestamps outside `tolerance_seconds`, malformed secrets/signatures, and mismatched signatures by raising `WebhookVerificationError`.

Add `tests/python-client.test.ts` using Vitest. The test should:
- Skip with a clear assertion failure only if Python is unavailable is not acceptable; instead detect `python` or `python3` and fail with a helpful message if neither exists.
- Start the in-process gateway with an in-memory store, create an app/API key through the existing TypeScript test helpers or storage APIs, and run Python snippets via `child_process.spawnSync`.
- Set `PYTHONPATH` to `clients/python`.
- Cover successful endpoint creation/listing, send/get/retry/list attempts/usage calls, endpoint delivery history/stats, and app-wide delivery listing filters.
- Cover one stable API error (`PosthornApiError` with status and code).
- Cover `verify_webhook` success, multi-signature success, and stale/bad-signature failures using deterministic vectors or signatures generated in the Python snippet.

Update the README Python SDK section so it reflects the implemented package and method surface. Do not claim the Python package is published to PyPI; document local/package usage only.

## Acceptance criteria (verbatim from roadmap/features.json)

1. A Python 3.9+ package exists under clients/python with PosthornClient, PosthornApiError, and verify_webhook using only the Python standard library.
2. PosthornClient supports tenant API calls for endpoint creation/listing, single and batch message sends, message status, message retry, message attempts, usage, endpoint delivery history/stats, and app-wide delivery listing with pagination/filter arguments.
3. Client errors expose HTTP status, stable Posthorn error code, and message without requiring callers to parse raw response bodies.
4. Python webhook verification accepts valid whsec_ Standard Webhooks signatures, supports multi-signature headers, enforces replay windows, and rejects stale, malformed, missing, or mismatched signatures.
5. Automated tests run the Python SDK against the in-process gateway and cover successful calls, API errors, pagination/filter query arguments, and signature verification failures.
6. README Python SDK documentation reflects the implemented methods and no longer points at a missing clients/python surface.
