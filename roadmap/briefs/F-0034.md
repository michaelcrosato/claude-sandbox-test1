# Builder Brief - F-0034: Message deduplication windows

> Immutable. Build exactly this; report back; nothing more. The source gap comes from
> `docs/PARITY.md`: Posthorn only has producer retry idempotency today, while comparable
> webhook gateways describe configurable duplicate suppression over a time window.

## Goal

Let producers suppress duplicate delivery fanout by supplying a deduplication key and a bounded
window on message intake.

This is not full arbitrary field extraction. Producers must provide the key explicitly.

## Product Rules

- New message fields:
  - `deduplicationKey`: optional string.
  - `deduplicationWindowSeconds`: optional positive safe integer.
- If `deduplicationKey` is omitted or `null`, deduplication is off and current behavior is unchanged.
- If `deduplicationKey` is present:
  - trim and validate it like a user-facing key: non-empty, no control characters, max 200 chars.
  - `deduplicationWindowSeconds` is optional; default to 24 hours.
  - valid window range is 1 second through 30 days.
  - compute `deduplicationExpiresAt = now + deduplicationWindowSeconds`.
- Duplicate definition:
  - same tenant app.
  - same `eventType`.
  - same `deduplicationKey`.
  - existing message has `deduplicationExpiresAt > now`.
- On a duplicate:
  - return the original accepted message and original fanout.
  - do not create a new message.
  - do not create new delivery rows.
  - do not increment accepted-message usage.
  - do not compare payloads or raise `idempotency_conflict`; deduplication intentionally suppresses noisy repeats.
- Existing `idempotencyKey` behavior wins first:
  - same `idempotencyKey` + same canonical body returns original result.
  - same `idempotencyKey` + different body still returns `idempotency_conflict`.
  - only new, non-idempotent requests should check deduplication.
- Deduplication is tenant-scoped and event-type-scoped.
- Expired dedup keys do not suppress new messages.
- Do not add arbitrary JSONPath, templates, user code, field extraction, endpoint-specific dedup rules, or background cleanup jobs in this feature.

## Expected Touch Points

- `src/storage.ts`: add nullable `deduplication_key` and `deduplication_expires_at` columns to
  `messages`, plus an index supporting app/event/key/expiry lookup and an idempotent migration.
- `src/messages.ts`: parse and store deduplication input, check active dedup records inside the
  existing transaction before quota/fanout/insert, and return the original fanout for duplicates.
- `src/openapi.ts`: add message write schema fields.
- `src/client.ts`: TypeScript SDK send-message and batch item input types.
- `src/cli.ts`: `posthorn client send ... [--deduplication-key <key>] [--deduplication-window-seconds <integer>]`.
- `clients/python/posthorn/client.py`: `send_message(..., deduplication_key=None, deduplication_window_seconds=None)`.
- `README.md`: API/SDK/CLI/Python docs and key capability.
- `docs/PARITY.md`: mark Posthorn deduplication rules as `Partial`, and say arbitrary field/window rules remain future work.

## Required Tests

- Storage:
  - new schema has the two columns and lookup index.
  - an existing legacy messages table gets the columns once.
- HTTP message intake:
  - duplicate same tenant/event/key inside the window returns the original message and fanout.
  - duplicate creates no new message, delivery rows, or accepted-message usage.
  - same key with a different event type is accepted as a new message.
  - same key in another tenant is accepted as a new message.
  - expired key is accepted as a new message.
  - malformed key/window values reject with `invalid_request`.
  - `idempotencyKey` conflict behavior remains unchanged when both fields are present.
- Batch:
  - duplicate items in one batch deduplicate against the first item.
  - mixed validation failures still return per-item results.
- Clients:
  - TypeScript SDK types include the fields.
  - CLI send supports the two options and redacts token-bearing usage errors.
  - Python SDK propagates the fields.
- Docs/contracts:
  - OpenAPI includes the fields.
  - README describes the feature without claiming arbitrary field extraction.
  - PARITY changes Posthorn deduplication rules from `Not yet` to `Partial`.
