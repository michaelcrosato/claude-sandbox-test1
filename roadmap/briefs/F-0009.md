# F-0009 Brief: Per-attempt audit log

## Scope

Expose delivery attempt history for a tenant-owned message through `GET /v1/messages/:id/attempts`.

The worker already records attempt rows in `delivery_attempts` with attempt number, outcome, response status, duration, failure reason, and attempted timestamp. This feature should add the tenant-scoped read path, pagination, and tests proving the API returns useful newest-first audit data without leaking another tenant's attempts.

## Acceptance Criteria

- Every delivery attempt records attempt number, outcome, timestamp, duration, response status when available, and failure reason when available.
- `GET /v1/messages/:id/attempts` returns a paginated newest-first audit log for the authenticated tenant.
- Tenants cannot read attempts for another tenant message.
- Tests cover success attempts, failed attempts, pagination, and tenant isolation.

## Implementation Notes

- Keep storage schema unchanged unless a defect is found; F-0007 already writes the required attempt details.
- Add a message-scoped listing helper that joins `delivery_attempts -> deliveries -> messages` and filters by `messages.app_id`.
- Return `404 not_found` for missing messages and other-tenant messages so message existence is not exposed.
- Use keyset pagination ordered by `attempted_at DESC, id DESC`.
- Keep route behavior consistent with existing gateway conventions: bearer auth first, JSON responses, `405 method_not_allowed` for unsupported methods, `400 invalid_request` for bad pagination input.
