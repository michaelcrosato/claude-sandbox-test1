# F-0014 Brief: TypeScript SDK and Command-Line Client

## Feature

Ship the first producer-facing client surface: a zero-dependency TypeScript SDK plus a `posthorn client` CLI for common tenant operations. The SDK and CLI must exercise the same HTTP contract users see, and must not call routes that are only aspirational.

## Acceptance

- `PosthornClient` can create/list/read/update/delete endpoints, send single and batch messages, fetch message status, retry dead-lettered messages, list delivery attempts, and read usage.
- The CLI supports `create-endpoint`, `send`, `list-endpoints`, `get-message`, `usage`, and `help`.
- Read commands print JSON to stdout. Mutating commands print useful JSON as well so scripts can consume secrets returned once.
- Client errors expose `status`, stable `code`, and message via `PosthornApiError`.
- SDK and CLI integration tests run against the in-process gateway and verify method-to-route coverage.

## Current Gap

F-0014 acceptance requires message status and retry, but the current HTTP API only has message send, batch send, attempts, and usage. Add the missing tenant routes as part of this feature:

- `GET /v1/messages/:id` returns `{ message, deliveries }`.
- `POST /v1/messages/:id/retry` resets tenant-owned `dead_letter` deliveries to `pending` and returns `{ retried }`.

Update README route status, OpenAPI implemented routes/schemas, and contract tests accordingly.

## SDK Surface

- `new PosthornClient({ baseUrl, apiKey, fetch? })`
- `listEndpoints(): Promise<{ data: EndpointRecord[] }>`
- `createEndpoint(input): Promise<CreateEndpointResult>`
- `getEndpoint(id): Promise<{ endpoint: EndpointRecord }>`
- `updateEndpoint(id, input): Promise<{ endpoint: EndpointRecord }>`
- `deleteEndpoint(id): Promise<void>`
- `sendMessage(input): Promise<AcceptMessageResult>`
- `sendMessageBatch(items): Promise<AcceptMessageBatchResult>`
- `getMessage(id): Promise<MessageStatusResult>`
- `retryMessage(id): Promise<RetryMessageResult>`
- `listMessageAttempts(id, options?): Promise<MessageAttemptsPage>`
- `getUsage(): Promise<{ usage: UsageSummary }>`

`PosthornApiError` carries:

- `status: number`
- `code: ApiErrorCode`
- `responseBody: unknown`

## CLI Surface

Environment:

- `POSTHORN_URL`
- `POSTHORN_API_KEY`

Commands:

- `posthorn client help`
- `posthorn client create-endpoint <url> [eventType...]`
- `posthorn client send <eventType> <jsonPayload> [--idempotency-key <key>]`
- `posthorn client list-endpoints`
- `posthorn client get-message <messageId>`
- `posthorn client usage`

All commands use stdout for successful output and stderr only for human-readable errors. CLI error output must not print API keys.

## Implementation Plan

1. Add message status and retry helpers in `src/messages.ts`.
2. Wire `GET /v1/messages/:id` and `POST /v1/messages/:id/retry` in `src/gateway.ts`.
3. Update `src/openapi.ts`, `README.md`, and existing OpenAPI/README sync tests for the new implemented routes.
4. Add `src/client.ts` with `PosthornClient`, public request/response types, and `PosthornApiError`.
5. Add `src/cli.ts` with zero-dependency argument parsing and an exported `runPosthornCli` test seam.
6. Export SDK/CLI types from `src/index.ts` and add package `main`, `types`, `exports`, `bin`, and `files` metadata without enabling publish.
7. Add SDK and CLI integration tests against an in-process gateway.
8. Run local product checks, evaluator, and security review.

## Risks and Guards

- API key handling is sensitive: Authorization headers must be set only from explicit config/env, and errors must not echo keys.
- New retry behavior mutates delivery state. Limit it to tenant-owned `dead_letter` deliveries and do not reset succeeded/pending/delivering rows.
- Keep the CLI dependency-free and deterministic. No shelling out from CLI tests.
- Package metadata should expose only runtime SDK/CLI outputs; do not ship compiled tests/scripts via `files`.
