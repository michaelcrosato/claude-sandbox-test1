# F-0017 Brief: Event Type Catalog and Consumer Portal Sessions

## Feature

Add the tenant event type catalog, synchronous endpoint test sends, and short-lived consumer portal sessions promised by the public route table. The feature must stay tenant-scoped, use the existing SQLite/no-dependency model, and avoid exposing reusable secrets or arbitrary outbound URL control.

## Acceptance

- Tenants can create, list, read, update, and archive event types with optional descriptions and optional JSON `schemaExample` payloads.
- Active event type names are unique per tenant; duplicate active creates return `409 conflict`.
- Archived event types do not appear in normal lists, cannot be read as active records, and are ignored by endpoint test-send fallback.
- `POST /v1/endpoints/:id/test` sends one synchronous signed webhook to the stored endpoint URL. It uses an explicit payload when supplied, otherwise uses the active event type's `schemaExample`, and reports `payloadSource: "explicit" | "schema_example"`.
- Disabled endpoint test sends return `400 endpoint_disabled` before making an outbound request.
- `POST /v1/portal/sessions` creates a short-lived, tenant-scoped portal token for endpoint management, stores only a hash, returns the raw token once, and exposes deterministic expiry behavior for tests.
- OpenAPI, README route rows, and README implemented error-code rows stay synchronized with runtime behavior.

## Contract

Event type routes:

- `GET /v1/event-types` -> `200 { data: EventType[] }`, active records only, newest first.
- `POST /v1/event-types` with `{ eventType, description?, schemaExample? }` -> `201 { eventType }`.
- `GET /v1/event-types/:id` -> `200 { eventType }` or tenant-scoped `404`.
- `PATCH /v1/event-types/:id` with `{ description?, schemaExample? }` -> `200 { eventType }`. Renaming `eventType` is intentionally not supported in this feature.
- `DELETE /v1/event-types/:id` archives the active record and returns `204`.

`EventType` shape:

- `id`, `eventType`, `description`, `schemaExample`, `archivedAt`, `createdAt`, `updatedAt`.
- `description` is `string | null`, max 500 chars, no control characters.
- `schemaExample` is a non-null JSON value with the same bounded JSON validation as message payloads.

Endpoint test-send route:

- `POST /v1/endpoints/:id/test` with `{ eventType, payload? }`.
- If `payload` is supplied, use it and return `payloadSource: "explicit"`.
- If `payload` is omitted, load the active catalog row for `eventType` and use its non-null `schemaExample`; return `payloadSource: "schema_example"`.
- Missing fallback example returns `400 invalid_request`.
- Success/failure response is synchronous and includes endpoint id, event type, payload source, delivery outcome, HTTP response status, duration, and bounded failure reason.
- It must use the stored endpoint URL, stored non-secret headers, Standard Webhooks signing, manual redirects, and configured gateway timeout.
- It must not insert `messages`, `deliveries`, `delivery_attempts`, or usage rows.

Portal sessions:

- `POST /v1/portal/sessions` with optional `{ endpointId?, expiresInSeconds? }`.
- Scope is fixed to `endpoint_management`.
- Default TTL is 15 minutes. Minimum 60 seconds. Maximum 24 hours.
- Response returns `{ session: { id, token, scope, endpointId, expiresAt, createdAt } }`.
- Persist only `token_hash`, never the raw token. Add an exported helper to validate/read active sessions by token for expiry tests and future dashboard use.

## Implementation Plan

1. Add shared JSON/event-type validation helpers where existing message and endpoint behavior can reuse them without broad refactors.
2. Extend SQLite schema with `event_types` and `portal_sessions`; update storage tests.
3. Add `src/event-types.ts`, `src/portal-sessions.ts`, and an endpoint test-send helper in `src/endpoints.ts` or a small dedicated module.
4. Extend `GatewayDependencies` with injectable delivery fetch for synchronous test sends and route `/v1/event-types`, `/v1/event-types/:id`, `/v1/endpoints/:id/test`, and `/v1/portal/sessions`.
5. Add OpenAPI routes/schemas/request bodies and update README route/error rows.
6. Add focused HTTP/unit tests for event type CRUD/archive/conflict, endpoint test-send explicit and schemaExample payloads, disabled endpoint behavior, portal token hashing/expiry, tenant isolation, and OpenAPI/README sync.
7. Run local checks, evaluator, and security review because this adds outbound delivery, token issuance, and new API surfaces.

## Risks and Guards

- Test-send SSRF: never accept a URL in the test-send body; only send to an already-validated stored endpoint URL.
- Token leakage: portal tokens are bearer credentials. Return raw token only on create, store only hashes, and keep tokens out of logs/spec examples.
- Error enum drift: add runtime coverage for `conflict` and `endpoint_disabled` so the closed OpenAPI enum remains honest.
- Data mutation: test sends are diagnostics, not message intake. Do not count them as accepted messages or create durable delivery tasks.
