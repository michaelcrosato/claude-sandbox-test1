# Builder Brief - F-0033: Endpoint payload delivery formats

> Immutable. Build exactly this; report back; nothing more. The source gap comes from `docs/PARITY.md`: Posthorn does not support payload transformations. This feature implements a safe declarative subset, not arbitrary JavaScript, templates, method changes, or URL rewrites.

## Goal

Let each endpoint choose the JSON body shape Posthorn sends:

- `envelope` (default): the current body, `{"id":"msg_...","eventType":"...","payload":{...}}`.
- `payload_only`: only the original message payload JSON.

This gives tenants a practical compatibility mode for receivers that already expect the event payload directly while preserving Posthorn's no-code-execution safety boundary.

## Product Rules

- Field name: `payloadFormat`.
- Stored values: `envelope` and `payload_only`.
- Default for new and existing endpoints: `envelope`.
- Create:
  - omitted or `null` means `envelope`.
  - valid strings are `envelope` and `payload_only`.
  - malformed values return `invalid_request`.
- Update:
  - omitted means unchanged.
  - `null` resets to `envelope`.
  - valid strings set the format.
  - malformed values return `invalid_request`.
- Read/list responses include `payloadFormat`.
- Signing must cover the exact raw body sent.
- Endpoint test-send must use the same body format as real deliveries.
- Do not add arbitrary code execution, templates, JSONPath, HTTP method rewrites, URL rewrites, or content-type customization.

## Expected Touch Points

- `src/storage.ts`: add `payload_format TEXT NOT NULL DEFAULT 'envelope'` to endpoints and an idempotent migration for existing DB files.
- `src/endpoints.ts`: types, create/update parsing, target rows, response serialization.
- `src/worker.ts`: select endpoint payload format, build delivery body from it, sign selected body.
- `src/endpoint-tests.ts`: build test-send body from endpoint payload format, sign selected body.
- `src/openapi.ts`: endpoint schema and endpoint write schema.
- `src/client.ts`: TypeScript SDK types.
- `src/cli.ts`: `posthorn client create-endpoint ... [--payload-format envelope|payload_only]`.
- `clients/python/posthorn/client.py`: `create_endpoint(..., payload_format=None)`.
- `README.md`: key capability/API/SDK/CLI/Python docs.
- `docs/PARITY.md`: mark Posthorn payload transformations as `Partial`, and update the gap text to say arbitrary transformations remain future work.

## Required Tests

- Storage migration:
  - new schema includes `payload_format`.
  - an existing legacy endpoints table gets `payload_format` defaulting to `envelope`.
- Endpoint HTTP:
  - create/read/list includes `payloadFormat`.
  - create with `payload_only` persists and returns it.
  - update can change to `payload_only` and reset with `null`.
  - invalid values reject on create and update.
- Worker:
  - default `envelope` preserves existing delivery body behavior.
  - `payload_only` sends only the original payload and verifies with the endpoint secret against that exact body.
- Endpoint test-send:
  - `payload_only` sends only the explicit/schema payload and signs that exact body.
- Clients:
  - TypeScript SDK create/update types include `payloadFormat`.
  - CLI create-endpoint supports `--payload-format`.
  - Python SDK create_endpoint propagates `payload_format`.
- Docs/contracts:
  - OpenAPI includes the field.
  - README describes it without claiming arbitrary transformations.
  - PARITY changes Posthorn payload transformations from `Not yet` to `Partial`.
