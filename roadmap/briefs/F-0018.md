# F-0018 Brief: Admin and Tenant Dashboard

## Feature

Add the first browser dashboard for Posthorn. The dashboard is served by the gateway, uses the same HTTP API routes as external clients, and makes tenant provisioning and delivery debugging possible without the CLI.

## Acceptance

- `GET /dashboard` serves an admin panel for listing tenants, creating tenants, minting API keys, revoking API keys, and viewing per-tenant usage.
- `GET /dashboard/tenant` serves a tenant panel for message history, message delivery status, per-attempt audit logs, endpoints, and usage.
- Dashboard flows use existing API routes where possible and show loading, empty, error, and success states.
- UI evidence includes browser screenshots for the main admin and tenant flows.

## Contract

Dashboard pages:

- `GET /dashboard` and `GET /dashboard/tenant` return `200 text/html`.
- Unsupported methods return `405 method_not_allowed`.
- The HTML routes do not embed admin tokens, tenant keys, hashes, endpoint secrets, or tenant data.
- Admin and tenant credentials are entered in the browser and sent only as `Authorization: Bearer ...` headers to API routes.
- Admin API routes remain disabled with `404` when `POSTHORN_ADMIN_TOKEN` is unset; the static dashboard can still load and show the API error in the panel.

Admin dashboard flow:

- List tenants with `GET /v1/admin/apps`.
- Create tenants with `POST /v1/admin/apps`.
- Mint tenant keys with `POST /v1/admin/apps/:id/keys`.
- List and revoke keys with `GET /v1/admin/apps/:id/keys` and `DELETE /v1/admin/keys/:id`.
- Read usage with `GET /v1/admin/apps/:id/usage`.
- Show a one-time secret reveal only from the key creation response; subsequent key lists must not expose reusable key material.

Tenant dashboard flow:

- List recent message history with `GET /v1/messages`.
- Read selected message delivery status with `GET /v1/messages/:id`.
- Read selected message attempt logs with `GET /v1/messages/:id/attempts`.
- Show tenant usage with `GET /v1/usage`.
- Show endpoints with `GET /v1/endpoints` for debugging context.

Message history route:

- Add `GET /v1/messages` because it is currently planned but required for real tenant history.
- Response is `{ data: Message[], nextCursor }`, tenant-scoped, newest first.
- Support `limit` and `cursor` only for this feature. Keep README filters such as `eventType`, `channel`, `after`, and `before` planned unless implemented.
- Use the same bounded pagination style as attempts.

## Implementation Plan

1. Add a small `src/dashboard.ts` module that returns static HTML/CSS/JS strings for the admin and tenant pages.
2. Route `/dashboard` and `/dashboard/tenant` in `src/gateway.ts` before API routing; serve as `text/html`.
3. Add `listMessages` to `src/messages.ts` with tenant-scoped keyset pagination and wire `GET /v1/messages` through the gateway.
4. Update OpenAPI and README so implemented routes match runtime behavior.
5. Add HTTP tests for dashboard pages, method handling, credential safety, state text, and dashboard API route references.
6. Add message-list HTTP tests for ordering, pagination, tenant isolation, auth, and method dispatch.
7. Capture browser screenshots for admin and tenant dashboard pages under `roadmap/evidence/F-0018/`.
8. Run local checks plus evaluator and security review because this touches API/auth-facing UI.

## Risks and Guards

- Credential leakage: never persist tokens in local storage or embed configured tokens in HTML. Keep credentials in memory in the browser session.
- Scope creep: do not add a frontend build system. The dashboard should stay dependency-free and served by the gateway.
- Message history privacy: `GET /v1/messages` must return only the authenticated tenant's messages and must not expose idempotency keys or payload hashes.
- Secret handling: API key secrets are displayed only after minting and are not present in list responses, OpenAPI examples, screenshots, or tests.
