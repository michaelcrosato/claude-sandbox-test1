# Builder Brief — F-0028: Admin command-line client

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped because this is a direct roadmap gap: `posthorn client` exists, but `posthorn admin` does not.

## Feature

Add a `posthorn admin` CLI namespace for common control-plane operations. It should use the existing HTTP admin API and `PosthornAdminClient`, print JSON to stdout, and keep admin credentials out of errors and logs.

This is an operator convenience layer only. Do not add new HTTP routes or new storage behavior.

## Command contract

Keep the existing tenant commands and backward-compatible `posthorn client ...` behavior.

Add:

```text
posthorn admin create-app <name> [--monthly-message-quota <integer|null>]
posthorn admin list-apps
posthorn admin get-app <appId>
posthorn admin update-app <appId> [--name <name>] [--monthly-message-quota <integer|null>]
posthorn admin delete-app <appId>
posthorn admin create-key <appId> [name]
posthorn admin list-keys <appId>
posthorn admin revoke-key <apiKeyId>
posthorn admin usage <appId>
posthorn admin rotate-system-secret <appId> [--overlap-seconds <integer>]
posthorn admin help
```

Environment:

- `POSTHORN_URL`
- `POSTHORN_ADMIN_TOKEN`

Output:

- Successful commands print JSON to stdout.
- `delete-app` and `revoke-key` can print a small JSON acknowledgement such as `{ "deleted": true }` / `{ "revoked": true }` because the HTTP APIs return 204.
- One-time secrets may appear only in successful `create-key` and `rotate-system-secret` stdout.
- Missing env, bad args, and API errors must return exit code 1, write one actionable stderr line, and not include `POSTHORN_ADMIN_TOKEN`.

Parsing:

- `--monthly-message-quota` accepts a non-negative safe integer or `null` to clear the quota.
- `--overlap-seconds` accepts a safe integer and lets the API enforce the configured bounds.
- Unknown options should fail with a usage error.
- `update-app` must require at least one field.

## Implementation notes

- Import and use `PosthornAdminClient` from `src/client.ts`.
- Add and export `POSTHORN_ADMIN_CLI_ROUTES`, mirroring `POSTHORN_CLI_ROUTES`.
- Keep tenant and admin route coverage separate in tests.
- Do not introduce a dependency for argument parsing.
- Do not weaken existing CLI assertions.

## Acceptance criteria (verbatim from roadmap/features.json)

1. posthorn admin help documents the admin commands and the POSTHORN_URL plus POSTHORN_ADMIN_TOKEN environment variables without requiring credentials.
2. Admin CLI commands cover common implemented admin routes: create/list/read/update/delete apps, create/list/revoke API keys, read app usage, and rotate app system signing secrets, printing JSON to stdout and one-time secrets only on create/rotate commands.
3. Missing environment, malformed inputs, and non-2xx API responses return a non-zero exit with actionable stderr that does not leak POSTHORN_ADMIN_TOKEN or one-time secret values.
4. Tests exercise the admin CLI against the HTTP gateway and pin its route map to implemented OpenAPI admin routes.
5. README and ROADMAP describe the implemented admin CLI honestly without claiming unimplemented commands.

## Suggested tests

- `tests/cli.test.ts`
  - `posthorn admin help` succeeds without env and documents admin commands/env vars.
  - Common admin operations against a gateway with `POSTHORN_ADMIN_TOKEN`: create app with quota, list app, read app, update app quota/name, create key, list keys, read usage, rotate system secret, revoke key, delete app.
  - Successful output does not contain `POSTHORN_ADMIN_TOKEN`.
  - Missing env, malformed quota/overlap, missing update fields, and API error cases return exit code 1 and do not leak the admin token.
  - `POSTHORN_ADMIN_CLI_ROUTES` equals the intended admin route set and each route exists in `IMPLEMENTED_ROUTES`.
- `tests/index.test.ts` or existing export coverage if needed
  - Package export includes the admin CLI route map if TypeScript coverage does not already catch it.
