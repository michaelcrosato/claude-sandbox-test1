# Builder Brief - F-0005: Standard Webhooks signing and verification utilities

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped because the upstream Standard Webhooks spec is small and directly verifiable from the primary spec document.

## Feature
Implement Standard Webhooks compatible signing and verification primitives from `README.md#receiving-webhooks`. This gives Posthorn a stable producer/consumer contract before delivery fanout exists: signed raw bodies, replay-window enforcement, and rotation-safe multi-signature headers.

## Spec
Current state:

- `src/index.ts` exports config, gateway, and storage primitives.
- `README.md#receiving-webhooks` already shows `verifyWebhook(endpointSecret, req.headers, rawBody)`, but the function is not implemented.
- Tests live under `tests/**/*.test.ts` and should import through `../src/index`.

Implement `src/webhooks.ts` with no runtime dependencies:

- Use built-in `node:crypto`.
- Export these constants:
  - `WEBHOOK_ID_HEADER = 'webhook-id'`
  - `WEBHOOK_TIMESTAMP_HEADER = 'webhook-timestamp'`
  - `WEBHOOK_SIGNATURE_HEADER = 'webhook-signature'`
  - `DEFAULT_WEBHOOK_TOLERANCE_SECONDS = 300`
- Export `createWebhookSecret(byteLength?: number): string`.
  - Default to 32 random bytes.
  - Allow only 24 through 64 bytes.
  - Return `whsec_` plus standard base64 bytes.
- Export `signWebhook(secretOrSecrets, rawBody, options?)`.
  - `secretOrSecrets` is a `string` or non-empty readonly string array.
  - `rawBody` is a `string`, `Buffer`, or `Uint8Array`; sign exactly these raw bytes.
  - `options.id` may provide the webhook id; otherwise generate a safe `msg_` id.
  - `options.timestampSeconds` may provide the integer Unix timestamp; otherwise use the current time in seconds.
  - Reject ids containing `.` and invalid timestamps, because the signed content is dot-delimited.
  - Decode each `whsec_` secret from base64 and use HMAC-SHA256.
  - Sign exactly `id + '.' + timestampSeconds + '.' + rawBody`.
  - Return an object with lowercase Standard Webhooks headers:
    - `webhook-id`
    - `webhook-timestamp`
    - `webhook-signature`
  - The signature header uses one or more space-delimited `v1,<base64-hmac>` values in the same order as `secretOrSecrets`.
- Export `verifyWebhook(secretOrSecrets, headers, rawBody, options?)`.
  - The signature matches the README call shape: `(secret, req.headers, rawBody)`.
  - Accept case-insensitive header names from plain records, Node-style `IncomingHttpHeaders`, or WHATWG `Headers`.
  - Require `webhook-id`, `webhook-timestamp`, and `webhook-signature`.
  - Reject ids containing `.`, malformed integer timestamps, malformed signature headers, stale timestamps, future timestamps outside tolerance, invalid secrets, and bad signatures.
  - `options.toleranceSeconds` defaults to `DEFAULT_WEBHOOK_TOLERANCE_SECONDS`; allow `0`, reject negative/non-integer values.
  - `options.nowSeconds` may pin the current timestamp for deterministic tests; otherwise use current time in seconds.
  - Parse space-delimited signatures and verify any `v1` symmetric signature against any supplied secret.
  - Use `crypto.timingSafeEqual` for digest comparison.
  - Return metadata `{ id, timestampSeconds }` on success.
- Export `WebhookVerificationError extends Error`.
  - Set `name = 'WebhookVerificationError'`.
  - Include a stable `code` field covering at least `missing_header`, `invalid_header`, `invalid_timestamp`, `timestamp_outside_tolerance`, `invalid_secret`, and `signature_mismatch`.
- Export useful TypeScript types for the options, returned headers, result, header input, and body input.
- Re-export the public API from `src/index.ts`.
- Update README implementation-status wording so it says Standard Webhooks signing/verification utilities are implemented.

Testing:

- Add `tests/webhooks.test.ts`.
- Generate deterministic synthetic `whsec_` values from fixed byte arrays in test helpers; do not commit real credentials.
- Cover deterministic signing:
  - Known secret bytes, raw JSON string, fixed id, fixed timestamp, expected exact headers, and expected exact `v1` signature.
  - A tampered body must fail verification.
- Cover verification failures:
  - Missing required headers.
  - Malformed timestamp.
  - Stale timestamp.
  - Future timestamp outside tolerance.
  - Bad signature.
  - Invalid `whsec_` secret format.
- Cover rotation/multi-signature behavior:
  - Signing with two secrets emits two space-delimited signatures.
  - Verification succeeds when only the second secret matches one of multiple signatures.
  - Unsupported `v1a` entries do not prevent a valid `v1` entry from succeeding.
- Cover body byte sensitivity with a raw `Buffer` or `Uint8Array`.

## Acceptance criteria (verbatim from roadmap/features.json)
1. Signing produces the expected Standard Webhooks headers for id, timestamp, and signature using whsec_ secrets.
2. Verification accepts valid raw-body signatures and rejects missing headers, malformed timestamps, stale timestamps, and bad signatures.
3. Replay-window behavior is configurable with a safe default.
4. Unit tests include deterministic vectors for valid, invalid, and multi-signature cases.
