# F-0021 Brief: App-Wide Delivery Listing and Filters

## Feature

Implement `GET /v1/deliveries` for tenant-scoped app-wide delivery search.

Operators should be able to find failed, pending, retrying, or endpoint-specific deliveries without starting from a message id. This completes the planned app-wide delivery listing route in README.

## Product Context

Adjacent webhook products make searchable delivery logs a core debugging surface:

- Hookdeck describes full observability into every event, including delivery status, failures, and performance: https://hookdeck.com/docs/use-cases/receive-webhooks
- Hookdeck lists monitoring delivery health, retry rates, dead-letter volume, response latency, and failure metrics as first-class webhook operations: https://hookdeck.com/docs/metrics
- Svix says its consumer portal lets customers add endpoints, pick event types, view logs, and replay deliveries without a Svix account: https://docs.svix.com/faq
- Stripe exposes an Event deliveries tab with Delivered/Pending/Failed states and delivery metadata for webhook endpoints: https://docs.stripe.com/webhooks
- Paddle documents finding failed webhook notifications by status and optional date window as the first step for recovery: https://developer.paddle.com/webhooks/about/respond-to-webhooks

Posthorn already has message history, endpoint history/stats, message attempts, and message-level retry. F-0021 adds the cross-app delivery list that operators expect when they know a receiver or failure class but not the original message id.

## Acceptance

- `GET /v1/deliveries` returns tenant-scoped deliveries across the app.
- Results are newest-first with keyset pagination by `deliveries.created_at DESC, deliveries.id DESC`.
- Response items include `id`, `messageId`, `endpointId`, `eventType`, `status`, `attemptCount`, `nextAttemptAt`, `lastError`, `createdAt`, and `updatedAt`.
- Response items do not include message payloads, endpoint URLs, endpoint headers, API keys, signing secrets, protected secret ciphertext/nonces, request bodies, or response bodies.
- Supported filters:
  - `status`: one of `pending`, `delivering`, `succeeded`, `dead_letter`.
  - `endpointId`: non-empty endpoint id string.
  - `eventType`: valid event type identifier.
  - `failureReason`: bounded non-empty string up to 100 characters; matches `deliveries.last_error` or any `delivery_attempts.failure_reason`.
  - `limit`: integer 1-100, default 50.
  - `cursor`: opaque cursor produced by previous response.
- Filtering by other-tenant endpoint id returns an empty list, not a 404 or leaked endpoint existence signal.
- Missing/invalid auth returns 401; unsupported methods return 405; invalid query params return `400 invalid_request`.
- OpenAPI and README mark `GET /v1/deliveries` implemented.
- Tests cover pagination, each filter, combined filters, tenant isolation, invalid queries, method/auth handling, and no payload/secret leakage.

## Contract

Route:

- `GET /v1/deliveries`
- Auth: tenant bearer key.

Query:

- `limit?: string`
- `cursor?: string`
- `status?: string`
- `endpointId?: string`
- `eventType?: string`
- `failureReason?: string`

Response:

```json
{
  "data": [
    {
      "id": "del_...",
      "messageId": "msg_...",
      "endpointId": "ep_...",
      "eventType": "invoice.paid",
      "status": "dead_letter",
      "attemptCount": 8,
      "nextAttemptAt": null,
      "lastError": "http_500",
      "createdAt": "2026-06-12T12:00:00.000Z",
      "updatedAt": "2026-06-12T12:05:00.000Z"
    }
  ],
  "nextCursor": null
}
```

Cursor payload:

- base64url JSON `{ "createdAt": "...", "id": "..." }`

## Implementation Plan

1. Add app-wide delivery listing helper, probably in a new `src/deliveries.ts` or by extending `src/endpoint-observability.ts` if reuse is clean.
2. Parse and validate filters with `MessageValidationError` or a small delivery-specific `invalid_request` error.
3. Query `deliveries` joined to `messages`, always scoped by `messages.app_id`.
4. Implement `failureReason` filter with `deliveries.last_error = ? OR EXISTS (...)`.
5. Wire `GET /v1/deliveries` in `src/gateway.ts`.
6. Add OpenAPI route and response schema.
7. Update README route table and SDK note.
8. Add focused HTTP tests.
9. Run evaluator and security review because this touches API/data surfaces.

## Risks and Guards

- Do not expose payloads or endpoint configuration through the list response.
- Keep all SQL parameterized; do not build filter values into SQL strings.
- Apply tenant scoping before every filter so other-tenant endpoint/message/delivery ids cannot leak.
- Keep `failureReason` bounded to prevent unhelpful broad scans or very large query strings.
