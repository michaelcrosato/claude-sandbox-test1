# F-0020 Brief: Endpoint Delivery History and Stats

## Feature

Implement endpoint-scoped delivery observability:

- `GET /v1/endpoints/:id/deliveries`
- `GET /v1/endpoints/:id/stats`

Tenants should be able to debug one receiver without scanning app-wide message history or correlating every message manually.

## Product Context

Competitor research points to per-destination observability as a core webhook product surface:

- Hookdeck documents delivery success rate, retry rate, dead-letter volume, response latency, queue depth, and per-destination monitoring as key webhook health metrics: https://hookdeck.com/docs/metrics
- Hookdeck also describes logs for requests, events, and delivery attempts plus replay/debug workflows in dashboard and API surfaces: https://hookdeck.com/docs/use-cases/receive-webhooks
- Convoy's manual highlights event delivery debugging, event filtering/search, and endpoint-targeted delivery structures: https://www.getconvoy.io/docs/product-manual/events-and-event-deliveries
- Convoy's README describes customer-facing dashboards where users can debug webhooks, retry events, add endpoints, and configure subscriptions: https://github.com/frain-dev/convoy

Posthorn already has durable deliveries, attempts, message history, endpoint CRUD, tenant dashboard basics, and Prometheus aggregate metrics. F-0020 should expose endpoint-focused API read models using the existing SQLite state.

## Acceptance

- `GET /v1/endpoints/:id/deliveries` returns tenant-scoped, newest-first delivery history for one endpoint.
- Delivery history is keyset-paginated with `limit` and `cursor`.
- Each delivery history item includes `id`, `messageId`, `endpointId`, `eventType`, `status`, `attemptCount`, `nextAttemptAt`, `lastError`, `createdAt`, and `updatedAt`.
- Delivery history does not include raw message payloads, endpoint URLs, endpoint headers, API keys, endpoint signing secrets, protected secret ciphertext, or request/response bodies.
- `GET /v1/endpoints/:id/stats` returns endpoint-scoped stats for a trailing window.
- Stats include `windowDays`, `since`, `until`, `total`, `byStatus`, `successRate`, `averageDurationMs`, `daily`, and `failureReasons`.
- `days` defaults to 7 and accepts safe integers from 1 to 90.
- Both routes require tenant bearer auth, return 404 for missing or other-tenant endpoints, and return 405 for unsupported methods.
- OpenAPI and README mark both routes implemented.
- Tests cover tenant isolation, pagination, stats math, empty endpoints, invalid query params, method/auth handling, and no secret/payload leakage.

## Contract

### `GET /v1/endpoints/:id/deliveries`

Query:

- `limit`: optional integer, default 50, min 1, max 100.
- `cursor`: optional opaque cursor produced by the previous response.

Response:

```json
{
  "data": [
    {
      "id": "del_...",
      "messageId": "msg_...",
      "endpointId": "ep_...",
      "eventType": "user.created",
      "status": "succeeded",
      "attemptCount": 1,
      "nextAttemptAt": null,
      "lastError": null,
      "createdAt": "2026-06-12T00:00:00.000Z",
      "updatedAt": "2026-06-12T00:00:01.000Z"
    }
  ],
  "nextCursor": null
}
```

Pagination order:

- `deliveries.created_at DESC, deliveries.id DESC`
- Cursor payload: base64url JSON `{ "createdAt": "...", "id": "..." }`

### `GET /v1/endpoints/:id/stats`

Query:

- `days`: optional integer, default 7, min 1, max 90.

Response:

```json
{
  "stats": {
    "endpointId": "ep_...",
    "windowDays": 7,
    "since": "2026-06-05T12:00:00.000Z",
    "until": "2026-06-12T12:00:00.000Z",
    "total": 10,
    "byStatus": {
      "pending": 1,
      "delivering": 0,
      "succeeded": 8,
      "dead_letter": 1
    },
    "successRate": 0.8,
    "averageDurationMs": 42,
    "daily": [
      {
        "date": "2026-06-12",
        "total": 2,
        "succeeded": 1,
        "failed": 1,
        "deadLettered": 1
      }
    ],
    "failureReasons": [
      { "reason": "http_500", "count": 1 }
    ]
  }
}
```

Stats semantics:

- `total` and `byStatus` count deliveries created within `[since, until]`.
- `successRate` is `succeeded / total`, or `null` when `total` is 0.
- `averageDurationMs` averages successful delivery attempt durations for deliveries created within the window, or `null` when no successful attempts exist.
- `daily.failed` counts failed attempts whose delivery was created within the window and whose attempted date falls on that UTC day.
- `daily.deadLettered` counts dead-letter attempts by attempted day.
- `failureReasons` groups non-null attempt failure reasons for deliveries created within the window.

## Implementation Plan

1. Add endpoint observability domain helpers, likely in a new `src/endpoint-observability.ts`.
2. Reuse existing endpoint tenant check before reading deliveries/stats.
3. Add keyset pagination parsing/encoding helpers local to the new module.
4. Wire `/v1/endpoints/:id/deliveries` and `/v1/endpoints/:id/stats` in `src/gateway.ts` before the generic endpoint-id route.
5. Add OpenAPI route entries and response schemas.
6. Update README route table and SDK note.
7. Add focused HTTP tests for history and stats.
8. Run evaluator and security review because the feature touches API/data surfaces.

## Risks and Guards

- Do not leak message payloads, endpoint URLs, endpoint headers, raw signing secrets, protected secret ciphertext, API keys, or response bodies.
- Keep endpoint existence checks tenant-scoped and before query-parameter validation where possible, so other-tenant endpoints remain 404.
- Keep stats derived from bounded windows to avoid accidental unbounded dashboard queries.
- Use deterministic UTC dates for daily buckets.
