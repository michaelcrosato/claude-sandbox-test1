# Builder Brief - F-0004: Tenant endpoint management CRUD

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped; the current storage, gateway, and Standard Webhooks modules are small enough to inspect directly.

## Feature
Implement the tenant-authenticated endpoint management routes from `README.md#api-routes`. Endpoints are the registered destinations that later message fanout and delivery workers will target.

## Spec
Current state:

- `src/gateway.ts` serves `/healthz` and `/readyz`.
- `src/storage.ts` already initializes `apps`, `api_keys`, and `endpoints` tables.
- `src/webhooks.ts` exports `createWebhookSecret()` for `whsec_` endpoint signing secrets.
- There is no admin API yet. F-0004 may add reusable API-key hashing/authentication helpers and tests may seed tenants directly in SQLite.

Implement tenant endpoint CRUD:

- Add dependency-free tenant API-key helpers.
  - API keys use the `phk_` prefix.
  - Export a generator and hash helper useful for tests and later admin work.
  - Authenticate `Authorization: Bearer phk_...` against `api_keys.key_hash`, ignoring revoked keys.
  - Return only tenant/app identity from auth; never echo API keys in responses.
- Add dependency-free endpoint store helpers.
  - Generate `ep_` endpoint IDs.
  - On create, generate a `whsec_` signing secret with `createWebhookSecret()`.
  - Return the secret only in the create response.
  - Persist a protected representation of the signing secret, not the plaintext. For this feature, a one-way `sha256:` digest is acceptable because delivery signing is not implemented yet.
  - List, fetch, update, and delete must always filter by authenticated tenant `app_id`.
  - Delete may hard-delete the row; deleted endpoints must be hidden from later reads/lists.
- Endpoint input:
  - `url`: required on create, optional on patch. Must be an absolute `http:` or `https:` URL, without credentials, and must not be localhost or a private/internal IP literal.
  - `eventTypes`: optional array of event type strings or `null`. `null`/omitted means all events. Event types must be non-empty full-stop-delimited identifiers made from letters, digits, and underscores.
  - `headers`: optional object of non-secret header names to string values. Reject invalid header names, control characters, and obvious sensitive/reserved headers such as `authorization`, `proxy-authorization`, `cookie`, `set-cookie`, `host`, `content-length`, and `x-api-key`.
  - `enabled`: optional boolean on patch.
- HTTP routes:
  - `GET /v1/endpoints`: bearer auth; return `200 { "data": [endpoint...] }`.
  - `POST /v1/endpoints`: bearer auth and JSON body; return `201 { "endpoint": {...}, "secret": "whsec_..." }`.
  - `GET /v1/endpoints/:id`: bearer auth; return `200 { "endpoint": {...} }` or `404`.
  - `PATCH /v1/endpoints/:id`: bearer auth and JSON body; return `200 { "endpoint": {...} }` or `404`.
  - `DELETE /v1/endpoints/:id`: bearer auth; return `204` or `404`.
  - Missing/invalid bearer auth returns `401` with `unauthorized`.
  - Malformed JSON returns `400` with `invalid_json`.
  - Validation failures return `400` with `invalid_request`, except private/internal URLs return `400` with `url_not_allowed`.
  - Unsupported methods on implemented endpoint paths return `405`.
- Endpoint response shape:
  - `id`, `url`, `eventTypes`, `headers`, `enabled`, `createdAt`, `updatedAt`.
  - Never include the signing secret or its persisted digest except as top-level `secret` on create.
- Re-export public auth/endpoint types/helpers from `src/index.ts` where useful for tests and later features.
- Update README implementation-status wording to mention tenant endpoint CRUD.

Testing:

- Add HTTP integration tests for endpoint CRUD against an in-memory gateway.
- Seed two synthetic tenants and API keys directly through storage helpers/SQL.
- Cover create/list/fetch/update/delete.
- Cover tenant isolation for list, fetch, update, and delete.
- Cover missing auth, revoked key auth, invalid URL, internal URL, invalid event types, and invalid/sensitive headers.
- Cover that create returns the `whsec_` secret once, later reads do not expose it, and the DB does not store the plaintext secret.

## Acceptance criteria (verbatim from roadmap/features.json)
1. Bearer-authenticated tenants can create endpoints with URL, event type filters, and optional headers.
2. Tenants can list, fetch, update, and delete only their own endpoints.
3. Created endpoints receive a whsec_ signing secret that is returned once and persisted hashed or otherwise protected from accidental disclosure.
4. Tests cover create/list/read/update/delete, tenant isolation, validation failures, and deleted endpoints being hidden from future reads.
