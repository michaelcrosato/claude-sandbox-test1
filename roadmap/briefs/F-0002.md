# Builder Brief - F-0002: Configuration and SQLite storage foundation

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out ran: one explorer mapped the current scaffold and file conventions; one confirmed `node:sqlite` availability and TypeScript import shape.

## Feature
Add Posthorn configuration loading and the default SQLite storage foundation from `README.md#configuration`. This enables later HTTP/API features to use one normalized config object and one idempotently initialized SQLite database, while preserving the single-container, no-runtime-dependency promise.

## Spec
Current scaffold:

- `src/index.ts` only re-exports `Gateway`, `GatewayConfig`, and `createGateway`.
- `src/gateway.ts` has a minimal `GatewayConfig` with `serviceName` and a frozen `createGateway` return shape.
- Tests live under `tests/**/*.test.ts` and import through the public entry point.

Build the F-0002 foundation:

- Add `src/config.ts` exporting `loadConfig(env?: NodeJS.ProcessEnv)` and config types.
- Parse the README configuration variables with defaults:
  - `POSTHORN_HOST`: default `0.0.0.0`
  - `POSTHORN_PORT`: default `3000`
  - `POSTHORN_DATA_DIR`: default `./posthorn-data`; allow `:memory:`
  - `POSTHORN_MAX_BODY_BYTES`: default `1000000`
  - `POSTHORN_ADMIN_TOKEN`: optional; when present, require at least 16 characters
  - `POSTHORN_WORKER_BATCH_SIZE`: default `16`
  - `POSTHORN_WORKER_CONCURRENCY`: default `8`
  - `POSTHORN_WORKER_REQUEST_TIMEOUT_MS`: default `10000`
  - `POSTHORN_WORKER_IDLE_POLL_MS`: default `1000`
  - `POSTHORN_WORKER_VISIBILITY_TIMEOUT_MS`: default `30000`
  - `POSTHORN_ENDPOINT_AUTO_DISABLE_AFTER_MS`: default `432000000`; allow `0`
- Reject invalid numeric values: non-integers, negative values where not allowed, zero where a positive value is required, and non-numeric strings.
- Add `src/storage.ts` using named imports from `node:sqlite`, preferably `DatabaseSync`.
- Implement file-backed and `:memory:` open/close behavior. For file-backed storage, ensure the data directory exists and open a stable SQLite file inside it.
- Implement an idempotent schema initialization path. Initial schema should create enough product-owned tables for upcoming features to extend: apps, api_keys, endpoints, messages, deliveries, delivery_attempts, and schema_migrations. Keep columns conservative and migration-safe; no live data assumptions.
- Expose a storage factory/type through `src/index.ts`.
- Update `createGateway` only as needed so `createGateway(loadConfig(...))` remains the intended public shape.
- Add `tests/config.test.ts` and `tests/storage.test.ts`, plus export smoke coverage if useful.
- Use only baseline `node:sqlite` APIs available in Node 24 CI and likely compatible with the README's Node 22.13 floor: `DatabaseSync`, `exec`, `prepare`, and `close`. Do not use newer optional SQLite APIs unless separately verified.
- Do not add runtime dependencies.
- Do not create `.env.example`; it is not in F-0002 scope.

## Acceptance criteria (verbatim from roadmap/features.json)
1. loadConfig parses documented POSTHORN_* environment variables with defaults and rejects invalid numeric values.
2. The storage layer opens and closes a SQLite database at POSTHORN_DATA_DIR or :memory: without external services.
3. A migration or schema initialization path creates the initial tables idempotently.
4. Unit tests cover defaults, invalid config, in-memory storage, and file-backed persistence using synthetic temp data.
