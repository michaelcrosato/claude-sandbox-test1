# F-0013 Brief: OpenAPI 3.1 Contract and Closed Error Codes

## Feature

Serve a machine-readable OpenAPI 3.1 document for every implemented public and admin route, and pin the public error-code envelope so runtime non-2xx responses cannot drift from the documented `Error.code` enum.

## Acceptance

- `GET /openapi.json` returns a valid OpenAPI 3.1 JSON document.
- The OpenAPI document includes every currently implemented route and method.
- Every non-2xx runtime API response uses `{ "error": { "code": "...", "message": "..." } }` with a stable code from the OpenAPI `Error.code` enum.
- Tests compare emitted runtime error codes to the OpenAPI `Error.code` enum.
- README implemented-route rows and OpenAPI routes are synchronized by test.

## Current Runtime Error Codes

- `invalid_request`
- `invalid_json`
- `url_not_allowed`
- `unauthorized`
- `not_found`
- `method_not_allowed`
- `idempotency_conflict`
- `payload_too_large`
- `quota_exceeded`
- `internal_error`

README currently lists planned future codes `endpoint_disabled` and `conflict`; keep them documented as planned, but do not include them in the runtime OpenAPI enum until code emits them.

## Implementation Plan

1. Add `src/openapi.ts` with a canonical implemented-route list, closed error-code enum, and `createOpenApiDocument()` generator.
2. Serve `GET /openapi.json` as an exact unauthenticated route in `src/gateway.ts`; non-GET on that path returns `405 method_not_allowed`.
3. Fix `/readyz` storage-not-ready and probe-failure `503` responses to use the standard error envelope.
4. Keep admin route auth ordering unchanged: disabled admin returns `404` before auth; enabled admin authenticates before method dispatch.
5. Update the README API table with a machine-readable status column and mark implemented routes as `implemented`; planned routes remain `planned`.
6. Update the README error table with a status column and mark runtime codes as `implemented`; planned-only codes remain `planned`.
7. Add `tests/openapi-contract.test.ts`:
   - fetch `/openapi.json` and check OpenAPI 3.1 shape.
   - compare OpenAPI operations to exported implemented routes.
   - parse README implemented route rows and compare to OpenAPI operations.
   - compare README implemented error codes to the OpenAPI `Error.code` enum.
   - trigger one runtime response for each emitted error code and compare the collected codes to the enum.

## Risks and Guards

- The OpenAPI route exposes admin route shapes intentionally. It must not expose configured admin state, tenant IDs, API keys, signing secrets, hashes, environment values, or local filesystem paths.
- Health/readiness are unauthenticated operational routes, but only non-2xx responses need the public error envelope for this acceptance.
- Keep the OpenAPI generator dependency-free and deterministic so tests can import it directly.
- Avoid documenting planned routes as implemented; the sync test should fail if README and OpenAPI drift.
