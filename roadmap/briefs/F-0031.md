# Builder Brief - F-0031: Helm chart deployment reference

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped because this is a direct roadmap gap: Docker Compose and monitoring artifacts exist, but the promised Helm chart does not. PostgreSQL scale-out remains a separate future backend feature and must not be claimed here.

## Feature

Add a starter Helm chart for running Posthorn on Kubernetes using the product's current single-container, SQLite-backed deployment model. The chart should make the happy path clear for operators while staying honest about current limits: one Posthorn pod, persistent `/data`, health/readiness probes, optional externally managed admin-token Secret, and no PostgreSQL/Redis/HA promises.

## Spec

Create `charts/posthorn/` with:

- `Chart.yaml`
- `values.yaml`
- `templates/_helpers.tpl`
- `templates/secret.yaml`
- `templates/pvc.yaml`
- `templates/deployment.yaml`
- `templates/service.yaml`
- `templates/NOTES.txt`

Chart behavior:

- Default release name should produce stable names through helpers.
- Default image repository: `posthorn`.
- Default image tag: chart `appVersion` or configurable `.Values.image.tag`.
- Default pull policy: `IfNotPresent`.
- Exactly one replica. Do not expose a replica count knob until the product has PostgreSQL-backed scale-out support.
- `POSTHORN_DATA_DIR` must be `/data`.
- Container port must be `3000`.
- Service type must be `ClusterIP` by default.
- Mount a PVC at `/data`; persistence enabled by default with a modest size such as `1Gi`.
- Health probe path: `/healthz`; readiness probe path: `/readyz`.
- Set `POSTHORN_ADMIN_TOKEN` from a Kubernetes Secret only. Support:
  - `.Values.admin.existingSecret` plus `.Values.admin.existingSecretKey`
  - or `.Values.admin.createSecret=true` with `.Values.admin.token` supplied by the installer
- Do not provide a default admin token. If chart-created secret is enabled without a token, the template should render a Helm `required` failure expression rather than a placeholder secret.
- Include sensible pod/container security context defaults such as `automountServiceAccountToken: false`, `runAsNonRoot`, `allowPrivilegeEscalation: false`, dropped capabilities, and `readOnlyRootFilesystem: true`; SQLite writes under the mounted `/data` volume.

Docs:

- Update `docs/DEPLOY.md` with a Helm chart section that explains:
  - create a Kubernetes Secret first, or pass `admin.existingSecret`
  - install from `charts/posthorn`
  - the chart is a starter reference for the current SQLite-backed single-pod deployment
  - PostgreSQL backend and multi-replica scale-out are not implemented yet
- Update `README.md` deployment mention to include Docker Compose and Helm.

Tests:

- Add `tests/helm-chart.test.ts` or extend deployment artifact tests.
- Tests should read chart files as text and assert:
  - all required files exist
  - Chart metadata identifies `posthorn`
  - deployment template includes `/healthz`, `/readyz`, `POSTHORN_DATA_DIR`, `/data`, port `3000`, and Secret-based `POSTHORN_ADMIN_TOKEN`
  - PVC and service templates exist with safe defaults
  - chart files do not contain `phk_`, `whsec_`, `POSTHORN_ADMIN_TOKEN=`, Redis, PostgreSQL, ServiceMonitor, HPA, or Ingress templates
  - docs mention `charts/posthorn`, `helm install`, `admin.existingSecret`, and the single-pod SQLite boundary

## Acceptance criteria (verbatim from roadmap/features.json)

1. A Helm chart exists under charts/posthorn with Chart.yaml, values.yaml, templates, and helpers for the Posthorn deployment, service, persistent volume claim, and admin-token secret wiring.
2. Default values deploy one Posthorn pod, persist /data, set POSTHORN_DATA_DIR=/data, expose port 3000 through a ClusterIP service, and configure health/readiness probes against implemented routes.
3. The chart avoids plaintext default secrets, supports an existing admin token secret, and does not render Redis, PostgreSQL, ingress, ServiceMonitor, or horizontal scaling features that the product does not yet support.
4. Deployment docs and README reference the Helm chart as a starter Kubernetes reference while keeping Docker Compose documented and PostgreSQL backend listed as future scale-out work.
5. Tests validate the chart files and templates contain the required Kubernetes resources, safe secret defaults, implemented probe paths, and no unsupported dependencies or secret-shaped sample values.
