# Builder Brief - F-0007: Crash-safe retry delivery worker

> Immutable. Explorer fan-out was skipped because F-0007 is tightly scoped to existing modules already read in this session: storage, endpoints, messages, gateway, Standard Webhooks helpers, README, and current tests.

## Feature
Add the delivery worker promised in `README.md#why-posthorn`: Posthorn must claim queued deliveries from SQLite, send signed HTTP webhooks, retry transient failures with exponential backoff, and eventually dead-letter tasks without Redis or any external broker.

## Spec
Implement an importable worker engine in `src/worker.ts` and export it from `src/index.ts`. The worker must:

- Claim eligible delivery rows from `deliveries` using a lease:
  - Eligible rows are `status = 'pending'` with `next_attempt_at` unset or due.
  - Reclaimable rows are `status = 'delivering'` with `lease_expires_at` due.
  - Claimed rows become `status = 'delivering'`, get a fresh `lease_expires_at`, and must not be claimed by a concurrent tick while the lease is active.
- Send one HTTP `POST` per claimed task to the endpoint URL.
- Build a deterministic JSON body from the stored message: `{"id": "...", "eventType": "...", "payload": ...}`.
- Sign the exact raw JSON body with the endpoint signing secret using `signWebhook`, adding `webhook-id`, `webhook-timestamp`, and `webhook-signature` headers.
- Add `content-type: application/json; charset=utf-8`, `user-agent: posthorn-delivery-worker`, and endpoint non-secret headers. Do not allow endpoint headers to override worker-managed headers.
- Treat 2xx responses as success. Mark the delivery `succeeded`, clear lease and last error, preserve attempt count, and insert a `delivery_attempts` row with outcome, attempt number, status, duration, and timestamp.
- Treat network errors, timeouts, and non-2xx responses as failures. Increment attempt count, insert a `delivery_attempts` row, and either:
  - reset to `pending` with exponential backoff in `next_attempt_at`, or
  - set `dead_letter` after the configured attempt budget is exhausted.
- Use a default attempt budget of 8 and a deterministic exponential backoff helper suitable for tests.
- Support aborting slow receivers at `requestTimeoutMs`.
- Return a small tick summary (`claimed`, `succeeded`, `failed`, `deadLettered`) for tests and embedding.

Endpoint signing secrets must become recoverable for new endpoints. F-0004 stored only a digest even though the schema already has ciphertext/key metadata fields. Add a small helper module that protects the returned `whsec_` value before persistence and reveals it for worker delivery. The persisted value must not contain the returned secret. Existing digest-only rows must fail closed: the worker should record a failed attempt/dead-letter path rather than sending unsigned traffic.

Do not implement the future public attempt-history API, metrics endpoint, auto-disable, admin provisioning, idempotency, dashboard, or SDK surfaces.

## Acceptance criteria (verbatim from roadmap/features.json)
1. The worker claims pending deliveries with a lease so concurrent ticks do not process the same task at the same time.
2. Successful HTTP delivery marks the task succeeded and records duration and response status.
3. Failed delivery schedules the next attempt using exponential backoff and dead-letters after the configured attempt budget.
4. Tests use a local synthetic receiver and cover success, retry, dead-letter, timeout, and lease reclaim behavior.
