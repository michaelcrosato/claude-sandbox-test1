# Builder Brief - F-0008: Idempotent message intake

> Immutable. Explorer fan-out was skipped because this feature is tightly scoped to the already-read message intake route, SQLite schema, and message HTTP tests.

## Feature
Add idempotency to `POST /v1/messages` as promised in `README.md#why-posthorn`. Producers retry network-failed sends; Posthorn must not duplicate message fanout when the same tenant resubmits the same request with the same idempotency key.

## Spec
Implement idempotency inside `src/messages.ts` and surface the documented conflict through `src/gateway.ts`.

Behavior:

- `POST /v1/messages` accepts an optional `idempotencyKey`.
- `idempotencyKey` must be a non-empty string after trimming, must not contain control characters, and must be at most 200 characters.
- For requests with an idempotency key, compute a stable SHA-256 fingerprint over the canonical request fields: `eventType` and `payload`.
  - Canonical JSON must sort object keys recursively so semantically equivalent parsed JSON objects hash the same way.
  - Do not include the idempotency key itself in the fingerprint.
- Store the key in `messages.idempotency_key` and the fingerprint in `messages.payload_hash`.
- The existing `UNIQUE(app_id, idempotency_key)` defines tenant scope; the same key may be reused independently by another tenant.
- If a tenant repeats the same key with the same fingerprint, return the original accepted message and the original delivery fanout IDs without inserting a new message or delivery rows.
- If a tenant repeats the same key with a different fingerprint, throw a typed idempotency conflict and return HTTP `409` with error code `idempotency_conflict`.
- Requests without an idempotency key must keep the current behavior and may create a new message on every call.

Do not implement batch intake, listing routes, retry routes, or OpenAPI changes in this feature.

## Acceptance criteria (verbatim from roadmap/features.json)
1. POST /v1/messages accepts an optional idempotencyKey and stores the request fingerprint per tenant.
2. A repeated request with the same key and same body returns the original accepted message without creating new delivery tasks.
3. A repeated request with the same key and different body returns the documented idempotency_conflict error.
4. Tests cover tenant scoping, same-body retry, changed-body conflict, and fanout count staying stable.
