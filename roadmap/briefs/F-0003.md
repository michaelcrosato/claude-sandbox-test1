# Builder Brief - F-0003: HTTP server with health and readiness endpoints

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out ran: one explorer mapped the current gateway/config/storage shape; one confirmed the zero-dependency Node HTTP/fetch testing approach.

## Feature
Build the first HTTP surface from `README.md#api-routes`: a startable/stoppable gateway with unauthenticated `/healthz` and `/readyz` endpoints. This turns the config and SQLite foundation into a running service that later API routes can extend.

## Spec
Current state:

- `src/gateway.ts` returns a frozen object with `serviceName` and preserved config only.
- `src/config.ts` already has `host`, `port`, `dataDir`, and body-size config.
- `src/storage.ts` has `openStorage()`, `listTables()`, raw `db`, and `close()`.
- Tests live under `tests/**/*.test.ts` and should import from `../src/index`.

Implement the HTTP gateway:

- Use built-in `node:http`; add no runtime dependencies.
- Extend `createGateway(loadConfig(...))` so the returned gateway has `start()` and `stop()` methods.
- `start()` should:
  - Open storage using `config.dataDir`.
  - Create a Node HTTP server.
  - Listen on `config.host` / `config.port`.
  - Return an address object with `host`, `port`, and `url` (or equivalent typed shape) that tests and future callers can use.
  - Be idempotent enough to reject or return the existing address on double-start; choose the simpler safe behavior and test it if exposed.
- `stop()` should close the HTTP server and close storage without leaking handles. Calling `stop()` before `start()` should be safe.
- Routes:
  - `GET /healthz`: return HTTP 200 with JSON and no auth. Include at least a stable `status: "ok"` field and service name.
  - `GET /readyz`: run a cheap storage probe, e.g. `SELECT 1` or `storage.listTables()`. Return HTTP 200 JSON when storage is reachable; return HTTP 503 JSON when the probe fails or storage initialization failed.
  - Unknown routes return the standard JSON error envelope shape if practical, or at minimum 404 JSON.
  - Non-GET methods on these paths may return 405 JSON.
- For a 503 readiness test, allow dependency injection through a small optional gateway dependency object, or another clean test seam, so the test can make the readiness probe throw without corrupting global state.
- Update `src/index.ts` only for new public gateway types.
- Update README implementation-status wording to mention health/readiness.

Testing:

- Add `tests/gateway-http.test.ts`.
- Use `loadConfig({ POSTHORN_HOST: '127.0.0.1', POSTHORN_DATA_DIR: ':memory:' })`, then override `port` to `0` in the config object for ephemeral test binding. `loadConfig` intentionally rejects `POSTHORN_PORT=0`, so do not weaken it.
- Use built-in global `fetch`.
- Always `await gateway.stop()` in `finally`.
- Cover successful start/stop, `/healthz` 200, `/readyz` 200 with in-memory storage, and `/readyz` 503 when the readiness probe fails.

## Acceptance criteria (verbatim from roadmap/features.json)
1. createGateway(loadConfig(...)) starts and stops an HTTP server without leaking handles.
2. GET /healthz returns 200 with a small JSON body and no authentication requirement.
3. GET /readyz returns 200 when storage can be queried and 503 when storage initialization fails.
4. Integration tests exercise both endpoints over HTTP against an in-memory store.
