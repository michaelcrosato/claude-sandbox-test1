# Builder Brief — F-0025: Auto-disable persistently failing endpoints

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped because this is a direct README/config contract gap: `README.md` documents endpoint auto-disable and `src/config.ts` already parses `POSTHORN_ENDPOINT_AUTO_DISABLE_AFTER_MS`, but the delivery worker never disables endpoints.

## Feature

Implement automatic endpoint disabling after a sustained delivery failure window. If an endpoint has been failing long enough to cross `endpointAutoDisableAfterMs`, Posthorn should set that endpoint's `enabled` flag to `false`, which already excludes it from future fanout and worker claims. Operators can disable this feature with `0`.

## Spec

Update the delivery worker to accept an `endpointAutoDisableAfterMs` option:

```ts
runDeliveryWorkerTick(storage, { endpointAutoDisableAfterMs: 432_000_000 })
createDeliveryWorker(storage, { endpointAutoDisableAfterMs: 432_000_000 })
```

Default it to the documented config value, `432_000_000` milliseconds. A value of `0` disables automatic endpoint disabling.

When a delivery is recorded as `dead_letter`, check whether the owning endpoint should be disabled. Disable it only when all of these are true:

- `endpointAutoDisableAfterMs > 0`
- the endpoint is still enabled
- there is at least one failed or dead-letter delivery attempt for that endpoint at or before the cutoff time (`now - endpointAutoDisableAfterMs`)
- there is no successful delivery attempt for that endpoint after that cutoff

Use existing durable tables. Do not add a migration unless the implementation truly needs one. Do not disable on retryable `failed` outcomes before a delivery reaches `dead_letter`.

Preserve current behavior:

- Existing pending messages already skip disabled endpoints through `acceptMessage`.
- Existing worker claims already filter `endpoints.enabled = 1`.
- Endpoint read/list APIs should continue to surface `enabled: false`.
- Do not leak payloads, endpoint headers, API keys, signing secrets, or protected secret metadata.

Wire the gateway endpoint test-send path or normal API route behavior only if needed. Do not introduce a background worker in this feature.

## Acceptance criteria (verbatim from roadmap/features.json)

1. Delivery worker options include endpointAutoDisableAfterMs, defaulting to the documented configuration value, with 0 disabling the behavior.
2. When a delivery dead-letters, the worker disables that endpoint only if the endpoint has had no successful delivery attempt since the auto-disable cutoff and has at least one failed or dead-letter attempt at or before the cutoff.
3. Auto-disabled endpoints are excluded from future message fanout and worker claims through the existing enabled=false behavior, while existing endpoint read/list APIs surface enabled=false without leaking payload or secret material.
4. Tests cover auto-disable after the configured failure window, no disable when the window is disabled, no disable when a recent success resets the failure window, and no disable for a different endpoint or tenant.
5. README configuration and key-capability text remain truthful for the implemented semantics.

## Suggested tests

Add focused coverage to `tests/worker.test.ts`:

- A delivery dead-letters after an old failed attempt and disables only that endpoint.
- `endpointAutoDisableAfterMs: 0` leaves the endpoint enabled after dead-letter.
- A successful attempt after the cutoff prevents auto-disable.
- A different endpoint with old failures is not disabled when the current endpoint dead-letters.
- After auto-disable, a new message to the same event type does not fan out to the disabled endpoint.

If implementation changes public exports or config plumbing, add the smallest matching test.
