# Builder Brief - F-0030: Endpoint delivery throttling

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out ran for endpoint CRUD/schema and worker claim behavior; the main finding was that the throttle belongs in `claimDeliveries` so not-yet-eligible work stays `pending`, while API/client surfaces need a normal nullable endpoint field.

## Feature

Add optional per-endpoint outbound delivery throttling. Official competitor docs show this as a standard webhook gateway control: Hookdeck destinations have a max delivery rate, Convoy configures egress rates per endpoint, and Svix supports endpoint throttling in messages per second. Posthorn should close the same reliability gap while staying simple: one nullable integer field named `rateLimitPerSecond`; no Redis, no concurrent-limit mode, no per-minute/per-hour modes.

## Spec

Implement a nullable endpoint field:

- Public JSON/TypeScript name: `rateLimitPerSecond`.
- Stored column: `endpoints.rate_limit_per_second INTEGER`.
- Meaning: `null` means no throttle; positive integer `N` means at most `N` delivery attempts for that endpoint may be claimed in each one-second endpoint claim window.
- Create default: omitted or `null` stores `NULL`.
- Update: omitted leaves unchanged; `null` clears the throttle; positive integer sets it.
- Validation: anything other than `null`, `undefined`, or a safe positive integer is `invalid_request`. Use the same error style as existing endpoint validation.

Storage:

- Add the column to `INITIAL_SCHEMA_SQL`.
- Add an idempotent migration helper similar to existing endpoint secret rotation columns so old SQLite databases get the column.
- Keep existing databases and tests working when the column is absent before initialization.

Endpoint domain/API:

- Extend `EndpointRecord`, `EndpointDeliveryTarget` row handling, `endpointFromRow`, `listEndpoints`, `getEndpoint`, and `getEndpointDeliveryTarget`.
- Extend `createEndpoint` and `updateEndpoint` parsing.
- Do not expose endpoint signing secret fields in read/list/update responses.
- Tenant isolation must remain the same: other tenants get 404 and must not learn whether a throttled endpoint exists.

Worker:

- Enforce throttling inside `claimDeliveries` before a delivery is leased.
- Candidate selection may over-fetch more than `batchSize` rows, but the returned/updated claims must never exceed `batchSize`.
- For an endpoint with `rateLimitPerSecond = N`, maintain internal endpoint throttle state in SQLite for the current one-second claim window. If the claim count is `>= N`, skip that endpoint for this tick and leave its deliveries untouched as `pending`.
- Multiple due deliveries for the same endpoint in the same tick count against the same durable allowance, so `N = 1` cannot claim two deliveries before any new attempt row exists.
- Unthrottled endpoints and other endpoints must still claim normally in the same tick.
- Existing retry due-time, expired lease reclaim, endpoint enabled, and at-least-once lease behavior remain unchanged.

OpenAPI/docs/clients:

- Add `rateLimitPerSecond` to the OpenAPI `Endpoint` schema and endpoint create/update request bodies where applicable.
- Update README examples/capabilities to mention optional endpoint delivery throttling without claiming unsupported modes.
- TypeScript SDK: extend `CreateEndpointInput` and `UpdateEndpointInput`.
- CLI: support `posthorn client create-endpoint <url> [eventType...] [--rate-limit-per-second <integer>]`, keep help current, and ensure malformed option errors do not echo user-provided secrets.
- Python SDK: extend `PosthornClient.create_endpoint(..., rate_limit_per_second=None)` and include the JSON field only when not `None`.

## Acceptance criteria (verbatim from roadmap/features.json)

1. Endpoint create and update accept an optional rateLimitPerSecond positive integer, allow null to clear it, validate malformed values with invalid_request, and read/list responses include the configured value without exposing secret material.
2. The storage schema persists the endpoint throttle setting for existing and new databases without breaking tenant isolation or existing endpoint behavior when the setting is unset.
3. The delivery worker respects each endpoint's configured per-second rate by leaving not-yet-eligible deliveries pending until the spacing window has elapsed, while other endpoints and unthrottled endpoints continue to be claimed normally.
4. OpenAPI, README, TypeScript SDK/CLI, and Python SDK documentation or helper surfaces reflect the implemented field without claiming unsupported throttle modes.
5. Tests cover create/update validation, read/list serialization, client/CLI propagation, worker throttling over time, and no throttling leak across endpoints or tenants.

## Suggested tests

- `tests/endpoints-http.test.ts`: create with `rateLimitPerSecond`, read/list include it, update clears with `null`, invalid create/update values return `invalid_request`, tenant isolation stays 404.
- `tests/storage.test.ts`: existing database migration adds `rate_limit_per_second`.
- `tests/worker.test.ts`: with `rateLimitPerSecond: 1`, first due delivery claims and records; second tick at `+500ms` claims zero and leaves the next delivery `pending`; tick at `+1000ms` claims it. Add a companion assertion that another endpoint is not blocked by the first endpoint's throttle.
- `tests/client.test.ts`, `tests/cli.test.ts`, `tests/python-client.test.ts`: prove the field is sent through helpers/CLI and does not leak credentials in malformed CLI output.
- `tests/openapi-contract.test.ts`: pin the schema if existing schema tests do not already catch the new field.
