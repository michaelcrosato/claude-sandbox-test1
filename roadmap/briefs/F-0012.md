# F-0012 Brief: Batch Message Intake

## Feature

Add `POST /v1/messages/batch` so producers can send 1 to 100 message objects in one authenticated request. The route returns per-item outcomes so invalid, conflicting, or quota-rejected items do not hide successful items in the same batch.

## Acceptance

- `POST /v1/messages/batch` accepts a top-level JSON array containing 1 to 100 message objects.
- Empty arrays, non-arrays, and arrays longer than 100 are top-level `400 invalid_request` errors.
- Missing auth, invalid JSON, and payload-too-large behavior matches existing tenant routes.
- Each valid item follows the same validation, idempotency, persistence, quota, usage, and fanout behavior as `POST /v1/messages`.
- The response status is `200` for a syntactically valid batch envelope and has `{ "results": [...] }`.
- Successful item results use `{ ok: true, message, fanout }`.
- Failed item results use `{ ok: false, error: { code, message } }` with existing stable codes such as `invalid_request`, `idempotency_conflict`, and `quota_exceeded`.
- Tests cover mixed success/failure batches, batch size boundaries, idempotency inside a batch, batch retry fanout stability, and quota partial success.

## Current Context

- Single-message intake lives in `src/messages.ts` `acceptMessage(storage, appId, input, now)` and already owns validation, idempotency, quota-before-fanout, persistence, delivery creation, and usage increments.
- `acceptMessage` returns an idempotent replay result before quota checks, which must remain true for batch sends.
- Gateway message routing currently handles `/v1/messages` and `/v1/messages/:id/attempts`; `/v1/messages/batch` is not routed.
- README implies `sendMessageBatch([ ... ])` and `batch.results.forEach((r) => { if (r.ok) ... else ... })`, but does not define extra fields.

## Implementation Plan

1. Add batch types and `acceptMessageBatch` in `src/messages.ts`.
2. Implement envelope validation separately from per-item processing. Only malformed batch envelopes throw `MessageValidationError`.
3. Process items sequentially through `acceptMessage`, using the same `now` value supplied by the gateway.
4. Convert item-level `MessageValidationError`, `MessageConflictError`, and `UsageQuotaExceededError` into per-item failure records. Unknown errors still throw.
5. Add exact `/v1/messages/batch` routing in `src/gateway.ts`, authenticate once, read one JSON body, and return `200` with the batch result.
6. Export the batch helper/types from `src/index.ts`.
7. Add focused HTTP tests for the route and quota tests for partial success with capped tenants.
8. Update README implementation status and optionally clarify the exact wire envelope now that it is implemented.

## Risks and Guards

- Do not reimplement message validation or fanout in the batch path; use `acceptMessage`.
- Top-level route errors should only cover the batch envelope, auth, invalid JSON, and body-size failures. Per-item validation/conflict/quota failures must not abort the batch.
- Sequential processing means earlier successful items commit before later failures. This is intentional partial-success behavior.
- Batch retry fanout stability depends on the existing idempotent replay path. Tests should add an endpoint between sends to prove replay does not re-match endpoints.
- Quota behavior should be natural and transparent: if only one quota slot remains, the first new item can succeed and later new items return per-item `quota_exceeded`; idempotent replays can still succeed.
