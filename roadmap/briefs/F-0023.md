# Builder Brief — F-0023: TypeScript SDK parity for implemented tenant routes

> Immutable. Build exactly this; report back; nothing more. Explorer fan-out was skipped because this is a direct contract gap from `README.md#typescript-sdk`: several implemented tenant HTTP routes are still missing typed TypeScript SDK helpers.

## Feature

Expand `PosthornClient` so TypeScript users can call every implemented tenant-facing route without dropping to raw HTTP. The goal is parity with the implemented OpenAPI tenant surface, not a new API contract. This should remove the README note that these helpers are planned for tenant routes.

## Spec

Update `src/client.ts`.

Add typed input/result shapes where useful and keep names idiomatic with the existing `PosthornClient` methods. Reuse exported product types from existing modules when available.

Required client helper methods:
- `listMessages(input?: { limit?: number; cursor?: string })`
- `rotateEndpointSecret(id: string)`
- `listEndpointDeliveries(id: string, input?: { limit?: number; cursor?: string })`
- `getEndpointStats(id: string, input?: { days?: number })`
- `listDeliveries(input?: { status?: DeliveryStatus; endpointId?: string; eventType?: string; failureReason?: string; limit?: number; cursor?: string })`
- `listEventTypes()`
- `createEventType(input: { name: string; description?: string | null; schemaExample?: JsonValue })`
- `getEventType(id: string)`
- `updateEventType(id: string, input: { description?: string | null; schemaExample?: JsonValue | null })`
- `deleteEventType(id: string)`
- `testEndpoint(id: string, input?: { eventType?: string; payload?: JsonValue })`
- `createPortalSession(input: { endpointId?: string; expiresInSeconds?: number })`

Update `POSTHORN_CLIENT_ROUTES` with every new method and route. The existing route coverage test must prove each route maps to `IMPLEMENTED_ROUTES`.

Update `tests/client.test.ts` to exercise the new helpers against the in-process gateway:
- Keep the existing SDK test intact.
- Add coverage for message listing pagination.
- Add endpoint secret rotation and assert the returned secret shape without exposing old secrets through normal reads.
- Add endpoint delivery history/stats and app-wide delivery listing after a synthetic delivery attempt.
- Add event type create/list/get/update/delete.
- Add endpoint test-send with an injected gateway delivery fetch so the test is local and deterministic.
- Add portal session creation.
- Add at least one new-helper error assertion, for example invalid event type creation conflict or invalid app-wide delivery filter.

Update `README.md`:
- Show representative TypeScript examples for message listing, endpoint observability, app-wide delivery listing, event type catalog, endpoint test-send, portal sessions, and secret rotation.
- Remove or narrow the note that TypeScript helpers are planned so it does not claim implemented helper routes are still missing.
- Keep admin/control-plane SDK claims unchanged unless the code already implements them.

Do not change gateway routes, OpenAPI route behavior, storage schema, scripts, or workflows unless a compile error demands a tiny type export adjustment inside `src/**`.

## Acceptance criteria (verbatim from roadmap/features.json)

1. PosthornClient exposes typed helpers for message listing, endpoint secret rotation, endpoint delivery history, endpoint stats, app-wide delivery listing with filters, event type CRUD, endpoint test-send, and portal session creation.
2. POSTHORN_CLIENT_ROUTES includes every new helper and the existing route coverage test proves each helper maps to an implemented OpenAPI route.
3. Client tests exercise representative successful calls for the new helpers against the in-process gateway, including pagination/filter query arguments where applicable.
4. Client tests cover at least one stable API error path for a new helper without weakening existing API error behavior.
5. README TypeScript SDK documentation reflects the implemented helper surface and no longer says these tenant helper routes are planned.
