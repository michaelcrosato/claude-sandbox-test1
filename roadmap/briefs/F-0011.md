# F-0011 Brief: Usage Metering and Quota Enforcement

## Feature

Track monthly tenant usage for accepted messages and delivery attempts, enforce per-tenant monthly message quotas before fanout, and expose the current-month usage read model to tenants and admins.

## Acceptance

- Accepted non-idempotent messages increment tenant message usage in the current UTC month.
- Replayed idempotent sends with the same request return the original result without incrementing usage or fanout again.
- Each recorded delivery attempt increments tenant delivery-operation usage in the month of the attempt timestamp.
- A tenant at or over its configured `monthlyMessageQuota` receives `429 quota_exceeded` before any new message or delivery task is persisted.
- `GET /v1/usage` returns the authenticated tenant's current-month counts and quota status.
- `GET /v1/admin/apps/:id/usage` returns the same read model for an admin-selected app, with missing apps returning `404`.
- Tests cover quota rejection, no fanout on quota rejection, delivery-attempt metering, admin usage reads, tenant auth isolation, and month-boundary reset.

## Current Context

- `apps.monthly_message_quota` already exists and admin app create/update parse it.
- Message intake lives in `src/messages.ts` in `acceptMessage`; it currently validates input, resolves matching endpoints before the transaction, checks idempotency inside `BEGIN IMMEDIATE`, then inserts `messages` and `deliveries`.
- Delivery attempt records are inserted in `src/worker.ts` only after the lease-state update succeeds in `recordSuccess` / `recordFailure`; stale workers roll back before inserting attempts.
- Admin routing in `src/gateway.ts` authenticates enabled admin routes before method dispatch. Preserve that property for new nested admin usage paths.
- Tenant routes authenticate per-handler through `authenticateTenantRequest`.
- `README.md` already documents `/v1/usage`, `/v1/admin/apps/:id/usage`, and `quota_exceeded`; update implementation status when runtime catches up.

## Implementation Plan

1. Add a `usage_months` SQLite table keyed by `(app_id, month)` with message and delivery-attempt counters.
2. Add `src/usage.ts` with helpers to compute UTC usage months, upsert counter increments, read a usage summary, and throw a quota error when message quota is exhausted.
3. Update `acceptMessage` so the idempotency fast path returns before quota checks, and new accepted messages assert quota and increment `messages_accepted` in the same transaction as message and delivery creation.
4. Update worker attempt insertion so a successful attempt audit insert also increments `delivery_attempts` in the same transaction, using the delivery's message/app relationship.
5. Add a gateway `now` dependency for deterministic month-boundary tests and use it for message intake plus current usage routes.
6. Add tenant and admin usage routes, preserving admin-disabled `404` behavior and admin auth before method dispatch.
7. Export public usage types/helpers from `src/index.ts`, update storage table tests, and add focused HTTP/worker tests.

## Risks and Guards

- Quota checks must run inside the message transaction to avoid concurrent over-acceptance.
- Idempotent retries must not be blocked by a quota reached after the original accepted send.
- Quota rejection must happen before inserting `messages` or `deliveries`.
- Admin route matching must not return `405` or `404` for known enabled admin usage paths before validating the admin token.
- Usage month is UTC `YYYY-MM`; month reset tests should inject a mutable clock through the gateway and worker options.
