# F-0016 Brief: Docker and Production Compose Reference

## Feature

Make the README's single-container deployment path real. Operators must be able to build a Posthorn image, run the compiled gateway with a persistent SQLite volume, set an admin token through environment configuration, and confirm `/healthz` without adding Redis or Postgres.

## Acceptance

- `Dockerfile` uses `npm ci` in a build stage, compiles TypeScript into `dist`, and runs the compiled server entrypoint in the runtime stage.
- The runtime container defaults `POSTHORN_DATA_DIR` to `/data`, exposes port `3000`, runs as a non-root user, and includes a `/healthz` healthcheck.
- `docker run` with a mounted data volume and `POSTHORN_ADMIN_TOKEN` starts the service and exposes `GET /healthz`.
- `docker-compose.yml` includes a single Posthorn app service, a persistent data volume, and a Prometheus scrape example for `/metrics`; it does not add Redis or Postgres.
- `docs/DEPLOY.md` gives copy/paste smoke commands for build, run, health, logs, shutdown, and compose.
- Tests statically pin the Dockerfile, compose reference, docs smoke commands, and the server entrypoint behavior. Real Docker smoke commands may be run locally when Docker is available, but CI should not require a Docker daemon.

## Implementation Plan

1. Add `src/server.ts` with a production process entrypoint that loads `loadConfig`, opens SQLite storage once, starts `createGateway`, starts the delivery worker on the same storage, and handles `SIGINT`/`SIGTERM`.
2. Export the server helper from `src/index.ts` so it is testable without spawning a long-running process.
3. Add `Dockerfile` with a build stage and a slim Node runtime stage using `node dist/src/server.js` as the default command.
4. Add `docker-compose.yml` with `posthorn`, a named `posthorn-data` volume, and a Prometheus service configured to scrape the app's `/metrics` endpoint.
5. Add `docs/DEPLOY.md` documenting exact `docker build`, `docker run`, `curl /healthz`, and compose commands. Use placeholders for secrets; never include a real token.
6. Add focused tests for the server helper and deployment artifacts. The artifact tests should parse the files as text and assert the contract that matters.
7. Run local product checks, then evaluator and security review because the feature touches deployment and secret-bearing environment configuration.

## Risks and Guards

- Docker image drift: static tests must pin `npm ci`, `npm run build`, compiled `dist/src/server.js`, `/data`, `USER node`, `/healthz`, and no Redis/Postgres services in compose.
- Secret handling: docs and compose must use placeholders or environment interpolation only. Do not commit a real admin token or API key.
- Worker/runtime split: the production server should run the delivery worker in-process using the same SQLite storage handle as the gateway, preserving the no-Redis single-container model.
- CI portability: do not require a Docker daemon for normal `npm test`; document real smoke commands and run them locally only when Docker is available.
